Lexer

clase TT

constantes TT (Token Types) Namespace de todas las constantes de tipo de token ›

Clase sin instancias que actúa como enumeración de cadenas. Cada atributo de clase es una cadena igual a su nombre (ej. TT.SENSOR == "SENSOR"). Esto permite usar las constantes tanto como valores en comparaciones como como etiquetas legibles en mensajes de error, sin necesidad de importar el módulo enum.

¿Por qué no usar enum.Enum? Las cadenas simples permiten serializar tokens a JSON directamente, y el parser puede usar los valores en set literales sin conversión. Las constantes de cadena también son directamente legibles en la salida de consola.

Agrupa los tokens en categorías: palabras reservadas, operadores lógicos, booleanos, valores discretos, dispositivos, atributos tipados, literales con unidad, literales simples, operadores y especiales. Ver la tabla de tokens para la lista completa.

Lexer

Tablas de clasificación

Estructuras de datos estáticas consultadas por el lexer durante la clasificación. No son funciones sino diccionarios y tuplas definidos a nivel módulo.

dict PALABRAS_RESERVADAS Mapea nombre en minúsculas → constante TT ›

Diccionario que permite clasificar en O(1) cualquier palabra que sea reservada. La clave se normaliza a minúsculas antes de la consulta, haciendo el lenguaje case-insensitive para palabras clave. Si la consulta no encuentra la clave, el identificador no es reservado y pasa a la siguiente etapa de clasificación.

PALABRAS_RESERVADAS["when"] → "WHEN"
PALABRAS_RESERVADAS["frio"] → "FRIO"
PALABRAS_RESERVADAS["xyz"] → KeyError (no existe)
tuple PREFIJOS_SENSOR / PREFIJOS_ACTUADOR Prefijos canónicos en orden de mayor a menor longitud ›

Tuplas ordenadas de mayor a menor longitud de prefijo para garantizar que la búsqueda encuentra primero el prefijo más específico. Por ejemplo, "sensor_movimiento" va antes que "sensor_temp" para que sensor_movimiento_sala no sea capturado parcialmente.

Orden importa: si "sensor_temp" estuviera antes que "sensor_movimiento", el identificador sensor_temp_exterior sería capturado correctamente por ambos de todas formas (por el chequeo startswith(pfx + "_")), pero el orden garantiza que en caso de ambigüedad prevalezca el más específico.
dict ATRIBUTOS_CONOCIDOS Mapea nombre de atributo → TT.ATTR_* ›

Diccionario consultado exclusivamente cuando el token anterior fue PUNTO. Si el nombre del atributo se encuentra, retorna el tipo específico (ATTR_BRILLO, ATTR_TEMP_OBJ, etc.); si no, retorna ATTR_GENERICO. Esto permite al parser escribir reglas de producción directas sin inspeccionar el valor del token.

ATRIBUTOS_CONOCIDOS["brillo"] → "ATTR_BRILLO"
ATRIBUTOS_CONOCIDOS["email_notif"] → "ATTR_EMAIL_NOTIF"
ATRIBUTOS_CONOCIDOS["xyz"] → (no existe) → ATTR_GENERICO
dict TABLA_RANGOS Define rangos válidos por combinación (dispositivo, atributo) ›

Diccionario cuya clave es la tupla (prefijo_canonico, TT.ATTR_*) y cuyo valor es (tipo_literal_esperado, rango_tupla, descripcion). Para sensores la clave usa None como atributo porque el valor aparece directamente en la condición. Es consultado por _validar_rango() para emitir advertencias semánticas tempranas.

TABLA_RANGOS[("sensor_temp", None)] → (TEMPERATURA, RANGO_TEMP_1, "sensor_temp")
TABLA_RANGOS[("aire_", ATTR_TEMP_OBJ)] → (TEMPERATURA, RANGO_TEMP_2, "aire_.temp_obj")
TABLA_RANGOS[("foco_", ATTR_BRILLO)] → (PORCENTAJE, RANGO_PERCENT, "foco_.brillo")

Helpers numéricos

Funciones auxiliares privadas que extraen el valor numérico de un token literal con unidad, para poder compararlo con los límites de rango.

func _num_temperatura(v) '25°C' → 25.0 | '-5°C' → -5.0 ›

Usa una expresión regular para capturar la parte numérica (con signo opcional y decimales opcionales) de un token TEMPERATURA y retorna un float. Ignora el sufijo °C.

_num_temperatura("25°C") → 25.0
_num_temperatura("-5.5°C") → -5.5
func _num_porcentaje(v) / _num_iluminancia(v) Extraen el número de '80%' y '600lux' respectivamente ›

Idénticas en estructura a _num_temperatura. Capturan la parte numérica descartando el sufijo de unidad.

_num_porcentaje("80%") → 80.0
_num_iluminancia("600lux") → 600.0
_EXTRACTOR: diccionario que mapea TT.TEMPERATURA → _num_temperatura, etc. Permite llamar al helper correcto sin un bloque if/elif en _validar_rango().
Lexer

clase SmartHomeLexer

Clase principal del análisis léxico. Mantiene el estado del análisis (contexto de dispositivo/atributo, último tipo visto) y expone una API pública de dos métodos.

método tokenizar(fuente, num_linea_base=1) Tokeniza un texto completo multilínea ›

Punto de entrada principal. Divide el texto en líneas preservando los saltos de línea, llama a _procesar_linea() por cada una, y agrega un token EOF al final. Reinicia errores, advertencias y el contexto al inicio de cada llamada.

ParámetroTipoDescripción
fuentestrTexto completo del archivo .smart
num_linea_baseintLínea desde la que comienza la numeración (default 1)
Retorna
List[Token] — todos los tokens excepto blancos y comentarios, más EOF al final
Efectos secundarios: popula self.errores con strings de error léxico y self.advertencias con strings de advertencia de rango.
método tokenizar_linea(linea, num_linea=1) Tokeniza una sola línea — modo interactivo ›

Variante de tokenizar() para el modo interactivo. Procesa una única línea sin reiniciar el contexto entre llamadas, permitiendo que la secuencia ACTUADOR → PUNTO → ATRIBUTO se mantenga aunque llegue en distintas llamadas.

Retorna
List[Token] — tokens de la línea más EOF
método _reset_ctx() Reinicia las tres variables de contexto a None ›

Pone a None los campos _ctx_prefijo, _ctx_attr_tipo y _ultimo_tipo. Se llama al inicio de tokenizar() y cuando el usuario escribe reset en modo interactivo.

self._ctx_prefijo = None # prefijo del disp. actual (ej. "foco_")
self._ctx_attr_tipo = None # TT.ATTR_* del último atributo visto
self._ultimo_tipo = None # TT del último token emitido
método _procesar_linea(linea, num_linea, tokens) Núcleo del análisis léxico: aplica el regex maestro caracter a caracter ›

Bucle principal del lexer. En cada iteración salta blancos y aplica _REGEX_MASTER.match(linea, pos) para encontrar el próximo token. Identifica qué grupo del regex capturó mediante m.lastgroup, obtiene el tipo desde _IDX_TO_TYPE, y llama a los métodos auxiliares. Si ningún patrón hace match, emite un DESCONOCIDO y agrega un error léxico.

while pos < len(linea): if espacio → pos++ m = _REGEX_MASTER.match(linea, pos) if not m → error léxico, DESCONOCIDO, pos++ else: tipo = _IDX_TO_TYPE[índice_del_grupo] if COMENTARIO → break if IDENTIFICADOR → _clasificar() tokens.append(Token(...)) _actualizar_ctx(tok) _validar_rango(tok) pos = m.end()
_REGEX_MASTER: un único objeto re.Pattern compilado con todos los patrones unidos por | en grupos numerados (?P<_0>…) | (?P<_1>…) | …. Esto garantiza que el orden de los patrones en la lista PATRONES determine la precedencia, y que el motor de regex elija la primera alternativa que coincida.
método _clasificar(palabra) Asigna la categoría correcta a un IDENTIFICADOR crudo ›

Se llama solo cuando el regex maestro capturó un IDENTIFICADOR. Aplica cuatro consultas en cascada con prioridad decreciente:

1. lower in PALABRAS_RESERVADAS → (TT.WHEN | TT.AND | TT.FRIO | …, UPPER) 2. lower.startswith(PREFIJO_SENSOR) → (TT.SENSOR, lower) 3. lower.startswith(PREFIJO_ACTUADOR) → (TT.ACTUADOR, lower) 4. self._ultimo_tipo == PUNTO → (ATRIBUTOS_CONOCIDOS.get(lower, ATTR_GENERICO), lower) 5. fallback → (TT.IDENTIFICADOR, palabra)
Atributos y el PUNTO: el paso 4 es la clave que permite emitir ATTR_BRILLO en lugar de IDENTIFICADOR. La condición _ultimo_tipo == TT.PUNTO garantiza que solo se active inmediatamente después de un punto, no en cualquier otro contexto.
método _prefijo_canonico(nombre) Extrae el prefijo canónico de un nombre completo de dispositivo ›

Dado el nombre completo de un sensor o actuador (ej. "aire_acondicionado"), retorna el prefijo que aparece como clave en TABLA_RANGOS (ej. "aire_"). Necesario porque los nombres de actuadores pueden tener sufijos arbitrarios dados por el usuario.

"sensor_temp_int" → "sensor_temp"
"foco_salon" → "foco_"
"aire_acondicionado" → "aire_"
"cerradura_principal" → "cerradura_"
método _actualizar_ctx(tok) Mantiene el autómata de contexto dispositivo→atributo ›

Actualiza las tres variables de contexto según el tipo del token recién emitido, implementando un autómata de estados simple:

SENSOR | ACTUADOR → _ctx_prefijo = prefijo_canonico(tok.valor) _ctx_attr_tipo = None ATTR_* → _ctx_attr_tipo = tok.tipo ATTR_GENERICO → _ctx_attr_tipo = None (sin rango definido) WHEN|IF|DO|END|… → reset completo del contexto (cualquier otro) → solo actualiza _ultimo_tipo
¿Por qué resetear con palabras reservadas de bloque? Al iniciar un nuevo WHEN, IF, etc., el contexto de dispositivo del bloque anterior ya no es relevante. Si no se resetea, el valor numérico del bloque siguiente podría ser validado contra el rango del dispositivo del bloque anterior.
método _validar_rango(tok) Emite advertencias semánticas tempranas por valores fuera de rango ›

Consulta TABLA_RANGOS con la clave (prefijo_actual, attr_tipo_actual). Si existe una entrada, verifica que el tipo del literal coincide y que el valor numérico (extraído con _EXTRACTOR) está dentro del rango. Si no, agrega una advertencia a self.advertencias. Solo procesa tokens de tipo TEMPERATURA, PORCENTAJE e ILUMINANCIA.

Nota teórica: la validación de rangos pertenece al análisis semántico, no léxico. Se incluye aquí como early semantic check, práctica habitual en compiladores reales cuando el contexto necesario ya está disponible en la fase léxica. Las advertencias no impiden que el token sea emitido ni que el análisis continúe.
func modo_interactivo() / modo_archivo(ruta) Puntos de entrada de la interfaz de usuario del lexer ›

modo_interactivo(): bucle que lee líneas desde stdin, llama a tokenizar_linea() y muestra los tokens. Soporta el comando reset para limpiar el contexto. Se mantiene el estado entre líneas para que instrucciones multilínea se tokenicen correctamente.

modo_archivo(ruta): valida existencia y extensión .smart, lee el archivo completo, llama a tokenizar(), imprime todos los tokens y el resumen de errores/advertencias. Sale con código 2 si hay errores léxicos.

ErrorSintactico

exception ErrorSintactico(Exception) Excepción lanzada cuando la gramática no se cumple ›

Excepción personalizada que transporta el mensaje de error y el token que lo provocó. Al convertirse a cadena incluye línea, columna, el mensaje y el token ofensivo. Es capturada en parsear() y _cuerpo() para implementar la recuperación de pánico.

str(error) → "Error sintactico | Linea 5, Col 12 | Se esperaba DO pero se encontro THEN (token: 'THEN' [THEN])"
Parser

clase Parser

Parser LL(1) descendente recursivo. Cada método corresponde a una regla de producción de la gramática.

método _actual() / _es(*tipos) Inspecciona el token actual sin consumirlo (lookahead) ›

_actual() retorna el token en la posición actual sin avanzar el cursor. _es(*tipos) es un helper booleano que retorna True si el token actual es de alguno de los tipos dados. Juntos implementan el lookahead de 1 símbolo que necesita un parser LL(1).

if self._es(TT.WHEN): # ¿el próximo token es WHEN? return self._bloque_when() # entonces parsear bloque WHEN
método _consumir(*tipos) Verifica el tipo del token actual, lo retorna y avanza el cursor ›

Operación fundamental del parser. Si el token actual es de uno de los tipos esperados, avanza el cursor y retorna el token. Si no, lanza ErrorSintactico con un mensaje que indica qué se esperaba y qué se encontró.

tok_when = self._consumir(TT.WHEN) # consume WHEN o lanza error tok_tiempo = self._consumir(TT.TIEMPO) # consume 30m, 1h, etc.
Diferencia con _avanzar(): _consumir() valida el tipo antes de avanzar; _avanzar() avanza incondicionalmente. Se usa _avanzar() en los lugares donde ya se sabe el tipo (ej. dentro de un bloque if self._es(TT.ELSE)).
método _avanzar() Avanza el cursor y retorna el token actual sin validar tipo ›

Usado cuando el tipo ya fue verificado por _es() en la línea anterior, por lo que repetir la validación sería redundante. También utilizado para consumir operadores AND/OR/NOT dentro de _condicion() y _expr_cond().

método _sincronizar(*tokens_de_seguimiento) Recuperación de pánico: descarta tokens hasta un punto seguro ›

Estrategia de recuperación de errores. Ante un ErrorSintactico, en lugar de abortar el análisis se descartan tokens hasta encontrar uno del conjunto de seguimiento (puntos donde el parseo puede retomarse con seguridad). Esto permite reportar múltiples errores en una sola pasada.

# En parsear(): sincronizar a inicio de nueva instrucción self._sincronizar(TT.WHEN, TT.EVERY, TT.IF, TT.END, TT.EOF) # En _cuerpo(): sincronizar a siguiente acción o cierre de bloque self._sincronizar(TT.ACTUADOR, TT.IF, TT.END, TT.ELSE, TT.EOF)
Técnica: conocida como panic mode recovery. Es la estrategia más simple de recuperación de errores y funciona bien para este lenguaje porque los bloques WHEN/EVERY/IF tienen delimitadores claros (END).
método parsear() Punto de entrada del parser — produce el NodoPrograma raíz ›

Corresponde a la regla programa ::= instruccion+ EOF. Itera llamando a _instruccion() hasta EOF. Captura los ErrorSintactico para aplicar recuperación de pánico y continuar el análisis.

Retorna
NodoPrograma — raíz del árbol de sintaxis abstracta
método _instruccion() Despacha al método correcto según el primer token (lookahead) ›

Regla: instruccion ::= bloque_when | bloque_every | condicional | asignacion. Usa el primer token para decidir qué regla aplicar. Si el token no es ninguno de los esperados, lanza ErrorSintactico.

WHEN → _bloque_when() EVERY → _bloque_every() IF → _condicional() ACTUADOR → _asignacion() otro → ErrorSintactico
método _bloque_when() WHEN condicion DO cuerpo END ›

Consume el token WHEN, parsea la condición, consume DO, parsea el cuerpo (lista de acciones), consume END. Retorna un NodoBloqueWhen con la condición y el cuerpo.

WHEN condicion DO accion+ END → NodoBloqueWhen
método _bloque_every() EVERY TIEMPO DO cuerpo END ›

Igual que _bloque_when() pero la "condición" es simplemente un token TIEMPO (ej. 30m). El valor del tiempo se guarda directamente en el nodo.

EVERY 30m DO accion+ END → NodoBloqueEvery(tiempo_valor="30m")
método _cuerpo() Parsea una lista de acciones hasta END, ELSE o EOF ›

Regla: cuerpo ::= accion+. Itera llamando a _accion() mientras el token actual no sea END, ELSE ni EOF. Implementa recuperación local de errores para poder continuar con la siguiente acción si una falla.

método _accion() Una acción es una asignación o un IF anidado ›

Regla: accion ::= asignacion | condicional. El token actual determina cuál: ACTUADOR → _asignacion(), IF → _condicional(). Permite anidar IFs dentro de WHEN y EVERY.

método _condicional() IF condicion THEN cuerpo [ELSE cuerpo] END ›

Parsea el bloque IF completo. El ELSE es opcional: si el token después del cuerpo THEN es ELSE, se parsea el cuerpo alternativo; si no, else_cuerpo queda como lista vacía. Retorna un NodoCondicional.

IF condicion THEN accion+ [ELSE accion+] END → NodoCondicional
método _condicion() Encadena expresiones simples con AND / OR ›

Regla: condicion ::= expr_cond ((AND|OR) expr_cond)*. Parsea la primera expresión simple y luego, mientras el token sea AND u OR, parsea otra expresión y crea un NodoCondicionCompuesta. Asocia a la izquierda de forma natural.

sensor_temp > 26°C AND alarma.estado == OFF → NodoCondicionCompuesta( izquierda = NodoCondicion(sensor_temp > 26°C), operador = "AND", derecha = NodoCondicion(alarma.estado == OFF) )
método _expr_cond() Expresión de condición simple: sensor/actuador op valor, o booleano puro ›

Parsea cuatro formas posibles de expresión condicional simple:

1. NOT expr_cond → NodoCondicion(negada=True) 2. TRUE | FALSE | ON | OFF → NodoCondicion(op=None) 3. SENSOR op_cmp valor_literal → NodoCondicion(izq=SENSOR) 4. ACTUADOR PUNTO ATTR op_cmp valor_literal → NodoCondicion(izq=ACTUADOR, izq_attr=...)
método _asignacion() ACTUADOR PUNTO ATTR = valor ›

Regla: asignacion ::= ACTUADOR PUNTO ATTR_* OP_ASIG valor_asig. Consume los cinco tokens en orden y construye un NodoAsignacion. El atributo debe ser un TT.ATTR_* (emitido por el lexer); si no lo es, _consumir() lanza error.

foco_sala.brillo = 80% → NodoAsignacion(actuador="foco_sala", attr_tipo=ATTR_BRILLO, attr_nombre="brillo", valor_tipo=PORCENTAJE, valor="80%")
método _valor_asig() / _valor_literal() Consumen el token del lado derecho de = o de una comparación ›

_valor_asig(): acepta cualquier token de TIPOS_VALOR más CADENA y EMAIL. Usado en el lado derecho de asignaciones donde puede haber texto o email.

_valor_literal(): igual pero sin CADENA ni EMAIL. Usado en condiciones donde esos tipos no tienen sentido comparar con operadores >, <, ==.

clase Recolector

Segundo pasaje sobre el AST. Transforma la estructura jerárquica del árbol en listas planas de InfoSeccion listas para ser renderizadas por el emisor HTML.

método recolectar(programa) Punto de entrada — itera las instrucciones raíz ›

Para cada instrucción del NodoPrograma llama a _procesar_instruccion() y acumula las InfoSeccion resultantes en una lista.

Retorna
List[InfoSeccion]
método _seccion_when / _seccion_every / _seccion_if(nodo) Crean un InfoSeccion para cada tipo de bloque ›

Cada método crea un InfoSeccion con el tipo correspondiente ("WHEN", "EVERY", "IF"), genera el texto del disparador llamando a _condicion_a_texto(), extrae los sensores de la condición, procesa el cuerpo de acciones, y para IFs con ELSE crea una subsección adicional.

método _volcar_asignacion(nodo, sec) / _obtener_actuador(sec, nombre) Agrupan atributos de un mismo actuador sin duplicar entradas ›

_obtener_actuador() busca en sec.actuadores si ya existe un InfoActuador con ese nombre; si no, lo crea y lo agrega. _volcar_asignacion() usa ese método y agrega la tupla (attr_nombre, valor_tipo, valor_raw) a la lista de atributos del actuador. Esto garantiza que todas las asignaciones de foco_sala queden en un único bloque HTML.

método _extraer_sensores_condicion(cond, sec) Recorre recursivamente la condición y extrae sensores únicos ›

Función recursiva que desciende por NodoCondicionCompuesta hasta llegar a NodoCondicion. Si el lado izquierdo es un SENSOR, lo agrega a sec.sensores (verificando que no esté duplicado). Esto permite renderizar el bloque verde del sensor junto al bloque del evento que lo dispara.

clase HtmlEmitter

Tercer y último pasaje. Recorre las InfoSeccion y genera el documento HTML según las reglas de la sección 5 de la consigna.

método emitir(secciones, titulo) Genera el documento HTML completo con cabecera, estilos y cuerpo ›

Construye la estructura HTML completa: <!DOCTYPE>, <head> con estilos CSS, y <body> con un <h1> de título. Luego itera las secciones llamando a _renderizar_seccion(). Retorna el HTML como string.

Retorna
str — documento HTML completo listo para guardar en disco
método _renderizar_seccion(sec, nivel) Genera el <div> de sección con título, sensores, actuadores y subsecciones ›

Para secciones de tipo "ASIG" (asignaciones sueltas) omite el envoltorio. Para el resto crea un <div class="seccion">, agrega el título disparador como <h2> o <h3> según el nivel de anidamiento, renderiza sensores, actuadores y subsecciones recursivamente.

método _renderizar_sensor(info, indent) → <div style="border:1px solid green; padding:20px"><h2>…</h2></div> ›

Genera el bloque visual del sensor según la consigna. Traduce el nombre técnico (sensor_temp) a un label legible (Temperatura) usando _LABEL_SENSOR, agrega la unidad desde _UNIDAD_SENSOR, y formatea el operador y el valor de la condición.

<div class="sensor" style="border:1px solid green; padding:20px; margin:10px 0;"> <h2>Temperatura > 26°C</h2> </div>
método _renderizar_actuador(info, indent) → <div gray><h1>nombre</h1><ul><li>…</li></ul></div> ›

Genera el bloque visual del actuador. El nombre va en <h1>, y cada atributo se convierte en un <li> dentro de <ul>. Itera la lista de atributos del InfoActuador llamando a _renderizar_item() para cada uno.

método _renderizar_item(label, valor_tipo, valor_raw, indent) Genera <li> normal o <li><a href="mailto:…"> para emails ›

Maneja los tres casos posibles del valor de un atributo. Para EMAIL extrae el nombre de usuario con split("@")[0] y genera un link mailto:. Para CADENA elimina las comillas del token. Para cualquier otro tipo usa el valor crudo directamente.

EMAIL → <li>Email de notificacion: <a href="mailto:user@dom.com">Contactar a user</a></li> CADENA → <li>Mensaje: texto sin comillas</li> otros → <li>Estado: ON</li>

función compilar()

func compilar(fuente, nombre_archivo) Orquesta las 4 fases del pipeline completo ›

Función de alto nivel que encadena las cuatro fases: instancia el SmartHomeLexer, luego el Parser, luego el Recolector, luego el HtmlEmitter, y retorna el resultado junto con los errores y advertencias de cada fase.

fuente → Lexer → tokens → Parser → AST → Recolector → secciones → HtmlEmitter → HTML
ParámetroTipoDescripción
fuentestrTexto completo del archivo .smart
nombre_archivostrNombre base para el título del HTML generado
Retorna
(html_str, errores_lexicos, advertencias_lexico, errores_sintacticos)
Diseño: separar la compilación de la E/S permite usar compilar() tanto desde modo_archivo() como desde modo_interactivo() y desde tests automáticos sin duplicar lógica.
