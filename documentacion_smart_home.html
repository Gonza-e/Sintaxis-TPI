General

Gramática con acciones semánticas

Las llaves { … } indican acciones semánticas: código que se ejecuta en el momento en que esa parte de la regla es reconocida. Esta notación es estándar en la teoría de compiladores.

programa ::= { instruccion }+ EOF { _emitir_cabecera() } { _emitir_cierre() } instruccion ::= bloque_when | bloque_every | condicional | asignacion_suelta bloque_when ::= WHEN condicion DO { _emit(<div seccion>) + _emit(<h2>Cuando:…</h2>) } { para cada sensor: _html_sensor() → <div green><h2>…</h2></div> } cuerpo END { _emit(</div>) } bloque_every::= EVERY TIEMPO DO { _emit(<div seccion>) + _emit(<h2>Cada: TIEMPO</h2>) } cuerpo END { _emit(</div>) } cuerpo ::= accion+ { mantiene OrderedDict { actuador: [<li>...] } } { al llegar a END/ELSE: _volcar_actuadores() } accion ::= asignacion | condicional condicional ::= IF condicion THEN { _emit(<div seccion>) + _emit(<h3>Si:…</h3>) } { para cada sensor: _html_sensor() } cuerpo [ ELSE { _emit(</div>) + _emit(<div seccion>) + _emit(<h3>En caso contrario</h3>) } cuerpo ] END { _emit(</div>) } asignacion ::= ACTUADOR PUNTO ATTR OP_ASIG valor_asig { acum[actuador].append(_html_item()) ← NO emite todavía } condicion ::= expr_cond ( (AND|OR) expr_cond )* { retorna (texto_legible, lista_de_sensores) — NO emite }

Lexer

clase TT

constantes TT (Token Types) Namespace de todas las constantes de tipo de token ›

Clase sin instancias que actúa como enumeración de cadenas. Cada atributo es una cadena igual a su nombre (TT.SENSOR == "SENSOR"). Permite usar las constantes en comparaciones, como claves de diccionario y como etiquetas legibles en mensajes de error sin importar enum.

En el parser de una pasada, TT se usa intensivamente en los conjuntos TIPOS_OP_CMP, TIPOS_ATTR y TIPOS_VALOR para los chequeos de lookahead con _es().

Lexer

Tablas de clasificación

dict / tuple PALABRAS_RESERVADAS / PREFIJOS_SENSOR / PREFIJOS_ACTUADOR / ATRIBUTOS_CONOCIDOS / TABLA_RANGOS Estructuras estáticas de clasificación consultadas durante la tokenización ›

PALABRAS_RESERVADAS: diccionario nombre_lower → TT.*. Permite clasificar en O(1) cualquier palabra reservada, booleano, modo o color.

PREFIJOS_SENSOR / PREFIJOS_ACTUADOR: tuplas ordenadas de mayor a menor longitud de prefijo para garantizar que el prefijo más específico se encuentre primero.

ATRIBUTOS_CONOCIDOS: diccionario nombre_lower → TT.ATTR_* consultado solo cuando el token anterior fue PUNTO. Permite al parser recibir tokens como ATTR_BRILLO en lugar de IDENTIFICADOR.

TABLA_RANGOS: diccionario (prefijo, TT.ATTR_*) → (tipo_esperado, rango, desc). Consultado por _validar_rango() para emitir advertencias de rango durante el análisis léxico.

Lexer

Helpers numéricos

func _num_temperatura(v) / _num_porcentaje(v) / _num_iluminancia(v) Extraen el valor numérico de un token literal con unidad ›

Funciones auxiliares que usan una expresión regular para capturar la parte numérica de tokens como "25°C", "80%" o "600lux" y retornan un float. Son usadas exclusivamente por _validar_rango() a través del diccionario _EXTRACTOR.

_num_temperatura("25°C") → 25.0 _num_porcentaje("80%") → 80.0 _num_iluminancia("600lux") → 600.0
Lexer

clase SmartHomeLexer

método tokenizar(fuente, num_linea_base=1) Tokeniza el texto completo — API principal del lexer ›

Divide el fuente en líneas y llama a _procesar_linea() por cada una. Reinicia errores, advertencias y el contexto al inicio. Agrega un token EOF al final.

Retorna
List[Token] — todos los tokens sin blancos ni comentarios, más EOF
método tokenizar_linea(linea, num_linea=1) Tokeniza una sola línea — modo interactivo ›

Variante para el modo interactivo. No reinicia el contexto entre llamadas, permitiendo que instrucciones distribuidas en múltiples líneas mantengan coherencia de contexto dispositivo→atributo.

método _reset_ctx() Reinicia _ctx_prefijo, _ctx_attr_tipo y _ultimo_tipo a None ›

Limpia las tres variables de estado que forman el autómata de contexto. Se llama al inicio de tokenizar() y cuando el usuario escribe reset en modo interactivo.

método _procesar_linea(linea, num_linea, tokens) Núcleo del análisis léxico — aplica el regex maestro ›

Bucle que aplica _REGEX_MASTER.match(linea, pos) en cada iteración. Si hay coincidencia, determina el tipo por el índice del grupo capturado, construye el token, y llama a _clasificar(), _actualizar_ctx() y _validar_rango(). Si no hay coincidencia, emite un token DESCONOCIDO y registra el error léxico.

_REGEX_MASTER: un único re.Pattern compilado con todos los patrones unidos por | en grupos numerados. El orden de los patrones en la lista PATRONES determina la precedencia, resolviendo posibles ambigüedades (ej. 250lux se captura como ILUMINANCIA antes de que el número 250 sea capturado como ENTERO).
método _clasificar(palabra) Asigna la categoría correcta a un IDENTIFICADOR crudo ›

Consulta en cascada con 5 niveles de prioridad:

1. lower in PALABRAS_RESERVADAS → (TT.WHEN | TT.AND | TT.FRIO | …) 2. startswith(PREFIJO_SENSOR) → TT.SENSOR 3. startswith(PREFIJO_ACTUADOR) → TT.ACTUADOR 4. _ultimo_tipo == PUNTO → ATRIBUTOS_CONOCIDOS.get(lower, ATTR_GENERICO) 5. fallback → TT.IDENTIFICADOR
El paso 4 es la clave que permite al parser recibir tokens específicos como ATTR_BRILLO o ATTR_EMAIL_NOTIF en lugar del genérico IDENTIFICADOR, habilitando las reglas de producción directas.
método _prefijo_canonico(nombre) Extrae el prefijo canónico de un nombre completo de dispositivo ›
"aire_acondicionado" → "aire_" "foco_salon" → "foco_" "sensor_temp_exterior" → "sensor_temp"
método _actualizar_ctx(tok) Autómata de contexto dispositivo → atributo ›

Mantiene las variables _ctx_prefijo y _ctx_attr_tipo actualizadas para que _validar_rango() sepa en qué contexto se encuentra el próximo literal.

SENSOR | ACTUADOR → guardar prefijo canónico, limpiar atributo ATTR_* → guardar tipo de atributo ATTR_GENERICO → limpiar atributo (sin rango definido) WHEN|IF|DO|END|… → reset completo del contexto
método _validar_rango(tok) Advertencias semánticas tempranas por valores fuera de rango ›

Consulta TABLA_RANGOS[(prefijo, attr_tipo)]. Si existe una entrada, extrae el número del token con _EXTRACTOR y lo compara con el rango. Emite una advertencia si está fuera. Solo actúa sobre tokens TEMPERATURA, PORCENTAJE e ILUMINANCIA.

Nota teórica: la validación de rangos pertenece al análisis semántico. Se incluye en el lexer como early semantic check porque el contexto necesario ya está disponible en esa fase. Las advertencias no detienen el análisis.
Parser

ErrorSintactico

exception ErrorSintactico(Exception) Se lanza cuando la gramática no se cumple ›

Excepción con dos datos: el mensaje descriptivo y el token que lo provocó. Al convertirse a cadena incluye línea, columna, mensaje y token. Es capturada en parsear() y _cuerpo() para implementar recuperación de pánico.

str(e) → "Error sintactico | Linea 5, Col 12 | Se esperaba DO pero se encontro THEN (token: 'THEN' [THEN])"
Parser

Conjuntos de lookahead

set TIPOS_OP_CMP / TIPOS_ATTR / TIPOS_VALOR Sets de constantes TT usados en _es() y _consumir() ›

Reemplazan los largos if self._es(TT.OP_EQ, TT.OP_NEQ, TT.OP_GT, …) por un único self._consumir(*TIPOS_OP_CMP). Su uso con el operador * (unpacking) hace el código más legible y fácil de extender.

También existe el diccionario auxiliar _OP_TEXTO que mapea cada tipo de operador a su símbolo legible (TT.OP_GTE → ">="), usado al construir el texto de los títulos de sección HTML.

Parser

clase ParserHTML

Clase central del parser de una pasada. A diferencia del parser anterior que producía un AST, esta clase produce directamente el HTML final. Su buffer interno self._out acumula las líneas HTML a medida que el análisis avanza.

método _actual() / _es(*tipos) Lookahead de 1 símbolo sin avanzar el cursor ›

_actual() retorna el token en la posición actual. _es(*tipos) retorna True si ese token es de alguno de los tipos dados. Son la base del lookahead LL(1): permiten decidir qué regla aplicar sin consumir el token.

método _consumir(*tipos) Verifica tipo, avanza y retorna el token — o lanza ErrorSintactico ›

Operación central del parser. Si el token actual es de los tipos esperados avanza y lo retorna. Si no, lanza ErrorSintactico con mensaje detallado. Es la operación más frecuente del parser.

self._consumir(TT.WHEN) # consume WHEN o lanza error self._consumir(TT.TIEMPO) # consume 30m, 1h, etc. self._consumir(*TIPOS_ATTR) # consume cualquier ATTR_*
método _avanzar() Avanza el cursor incondicionalmente y retorna el token ›

Usado cuando el tipo ya fue verificado por _es() justo antes, por lo que repetir la validación sería redundante. Ejemplo típico: dentro de if self._es(TT.ELSE): self._avanzar().

método _sincronizar(*tipos_seguimiento) Recuperación de pánico: descarta tokens hasta un punto seguro ›

Estrategia de recuperación de errores. Descarta tokens hasta encontrar uno del conjunto de seguimiento, permitiendo continuar el análisis y reportar múltiples errores en una sola ejecución.

Panic mode recovery: la estrategia más simple de recuperación. Funciona bien para este lenguaje porque WHEN, EVERY, IF y END son delimitadores claros de instrucciones.
método _emit(linea) Agrega una línea al buffer de salida HTML ›

El método más simple del parser: hace self._out.append(linea). Es la única forma de producir salida HTML. Al final de parsear(), el buffer se une con "\n".join(self._out) para producir el HTML completo.

¿Por qué un buffer en lugar de escribir directo al archivo? Usar un buffer en memoria permite retornar el HTML como string desde parsear(), lo que facilita el testing y el modo interactivo (donde se imprime en consola en lugar de guardar a disco).
método _html_sensor(nombre, op, valor, ind) Acción semántica: emite bloque verde de sensor ›

Genera el bloque HTML del sensor según la consigna (borde verde 1px, padding 20px). Traduce el nombre técnico a legible usando _LABEL_SENSOR, agrega la unidad desde _UNIDAD_SENSOR, y formatea el operador con _OP_TEXTO.

_html_sensor("sensor_temp", TT.OP_GT, "26°C", " ") → emite: <div style="border:1px solid green; padding:20px; margin:10px 0;"> <h2>Temperatura > 26°C°C</h2> </div>
método _html_actuador(nombre, items, ind) Acción semántica: emite bloque gris completo de actuador ›

Genera el bloque HTML completo del actuador según la consigna (borde gris 1px, padding 20px). Recibe la lista de strings <li> ya construidos y los emite todos de una vez dentro de un <ul>. Este método no sabe cuántos atributos hay ni cuándo llegaron — simplemente vuelca lo que _cuerpo() acumuló.

_html_actuador("foco_entrada", ["<li>Estado: ON</li>", "<li>Brillo: 80%</li>"], " ") → emite: <div style="border:1px solid gray; padding:20px; margin:10px 0;"> <h1>foco_entrada</h1> <ul> <li>Estado: ON</li> <li>Brillo: 80%</li> </ul> </div>
método _html_item(attr_nombre, valor_tipo, valor_raw) Construye el string <li> — NO lo emite todavía ›

Maneja los tres casos del valor de un atributo y retorna el string <li> sin emitirlo. El llamador (_asignacion()) lo deposita en el acumulador.

EMAIL → <li>Email notif: <a href="mailto:user@dom.ar">Contactar a user</a></li> CADENA → <li>Mensaje: texto sin comillas</li> otros → <li>Estado: ON</li>
★ clave _volcar_actuadores(acumulador, ind) Vuelca todos los actuadores acumulados y limpia el dict ›

Itera el OrderedDict y llama a _html_actuador() por cada entrada, luego limpia el dict con .clear(). Es el momento en que las asignaciones acumuladas se convierten en HTML real.

¿Por qué este método es clave?
Sin él, sería imposible agrupar múltiples asignaciones del mismo actuador en una sola pasada. El patrón es: las asignaciones recuerdan en lugar de emitir, y este método es el que finalmente convierte lo recordado en HTML. Se llama en dos momentos:
1. Antes de entrar a un IF anidado (para que los actuadores aparezcan antes de la subsección)
2. Al salir del bucle de _cuerpo() al encontrar END o ELSE
acum = {"foco_sala": ["<li>Estado: ON</li>", "<li>Brillo: 80%</li>"], "aire_ac": ["<li>Modo: FRIO</li>"]} _volcar_actuadores(acum, " ") → emite bloque gris de foco_sala con 2 atributos → emite bloque gris de aire_ac con 1 atributo → acum queda vacío
método parsear() Punto de entrada — emite cabecera, parsea programa, emite cierre ›

Método público principal. Llama a _emitir_cabecera(), itera instrucciones hasta EOF con recuperación de pánico ante errores, llama a _emitir_cierre(), y retorna el HTML completo como string.

Retorna
str — documento HTML completo listo para guardar a disco
método _emitir_cabecera() / _emitir_cierre() Emiten el DOCTYPE, head, body de apertura y el cierre del documento ›

_emitir_cabecera() genera todo el encabezado HTML: <!DOCTYPE html>, metadatos, título, bloque <style> con los estilos CSS, y el <h1> del dashboard. Se llama una única vez al inicio de parsear().

_emitir_cierre() genera simplemente </body></html>. Se llama una única vez al final.

método _instruccion(nivel) Despacha al método correcto según el lookahead ›

Regla: instruccion ::= bloque_when | bloque_every | condicional | asignacion_suelta. Usa el token actual para decidir. El caso de asignación suelta en nivel raíz crea su propio OrderedDict local y lo vuelca inmediatamente.

WHEN → _bloque_when(nivel) EVERY → _bloque_every(nivel) IF → _condicional(nivel) ACTUADOR → acum={}; _asignacion(acum); _volcar_actuadores(acum) otro → ErrorSintactico
método _bloque_when(nivel) WHEN condicion DO cuerpo END — con acciones semánticas intercaladas ›

Parsea la regla completa intercalando acciones semánticas de emisión HTML. La condición se parsea antes de emitir el <div> porque necesitamos el texto de la condición para el <h2> — _condicion() retorna ese texto sin emitir nada.

consumir WHEN (texto, sensores) ← _condicion() ← NO emite HTML consumir DO _emit(<div seccion>) ← acción semántica _emit(<h2>Cuando: {texto}</h2>) ← acción semántica para sensor en sensores: _html_sensor() ← acción semántica _cuerpo(nivel+1) ← regla recursiva consumir END _emit(</div>) ← acción semántica
método _bloque_every(nivel) EVERY TIEMPO DO cuerpo END ›

Idéntico en estructura a _bloque_when() pero más simple: el "disparador" es el token TIEMPO directamente, sin necesidad de parsear una condición. El valor del tiempo se incluye en el <h2>.

★ clave _cuerpo(nivel) Núcleo del agrupamiento en una pasada — mantiene el OrderedDict acumulador ›

Este método es el corazón de la técnica de una sola pasada. La clave está en el OrderedDict local que vive solo durante la ejecución de este método:

El ciclo de vida del acumulador:
— Se crea vacío al inicio de _cuerpo()
— Cada _asignacion() deposita un <li> en acum[nombre_actuador]
— Si aparece un IF, primero se vuelcan los actuadores pendientes (para mantener el orden visual correcto) y luego se parsea el IF
— Al encontrar END o ELSE, se vuelcan todos los actuadores restantes
— Al retornar, el acumulador se destruye automáticamente (es una variable local)
acum = OrderedDict() # vive solo mientras dura este cuerpo mientras NO (END | ELSE | EOF): si IF: _volcar_actuadores(acum, ind) ← vaciar antes del IF _condicional(nivel) ← parsear el IF (emite su propio HTML) si ACTUADOR: _asignacion(acum) ← acumular, NO emitir si otro: ErrorSintactico + _sincronizar() # al salir del bucle: _volcar_actuadores(acum, ind) ← emitir lo que quedó pendiente
¿Por qué OrderedDict y no dict? En Python 3.7+ los dict ordinarios ya mantienen el orden de inserción, por lo que OrderedDict y dict se comportan igual. Se usa OrderedDict explícitamente para dejar claro en el código que el orden de aparición de los actuadores es intencional y debe preservarse en el HTML.
método _condicional(nivel) IF condicion THEN cuerpo [ELSE cuerpo] END ›

Parsea el bloque IF intercalando emisión HTML. El ELSE es opcional. El parámetro nivel determina si el título usa <h2> (nivel raíz) o <h3> (anidado dentro de WHEN/EVERY).

consumir IF (texto, sensores) ← _condicion() consumir THEN tag = "h3" si nivel > 0 else "h2" _emit(<div> + <{tag}>Si: {texto}</{tag}>) para sensor: _html_sensor() _cuerpo(nivel+1) ← crea su propio acum interno si ELSE: _emit(</div>) _emit(<div> + <{tag}>En caso contrario</{tag}>) _cuerpo(nivel+1) ← nuevo acum para el cuerpo ELSE consumir END _emit(</div>)
método _condicion() Encadena expresiones con AND/OR — retorna datos, NO emite ›

Regla: condicion ::= expr_cond ((AND|OR) expr_cond)*. Acumula el texto legible y la lista de sensores mencionados a medida que parsea. El HTML no se emite aquí: el llamador recibe los datos y decide cuándo y dónde emitirlos (con la indentación correcta).

Retorna
(texto: str, sensores: list[tuple(nombre, op_tipo, valor)])
¿Por qué no emitir aquí? Porque en el momento en que se parsea la condición, el método llamador aún no emitió el <div> contenedor. Si _condicion() emitiera el bloque del sensor, aparecería antes del <div> de sección, generando HTML inválido.
método _expr_cond() Expresión simple: sensor op valor, actuador.attr op valor, o booleano ›

Parsea una de cuatro formas posibles y retorna el texto legible + sensores encontrados. Cuando el lado izquierdo es un SENSOR, registra la tupla (nombre, op_tipo, valor) en la lista de sensores para que el llamador emita el bloque verde.

NOT expr_cond → ("NOT {texto}", sensores) TRUE | FALSE | ON | OFF → ("TRUE", []) SENSOR op valor → ("sensor_luz < 250lux", [("sensor_luz", OP_LT, "250lux")]) ACTUADOR PUNTO ATTR op valor → ("aire_ac.estado == OFF", []) ← sin bloque verde
★ clave _asignacion(acum) Acumula en el dict — NO emite HTML directamente ›

Esta es la diferencia más importante respecto a un parser naïve de una pasada. En lugar de llamar a _emit(), construye el string <li> con _html_item() y lo deposita en acum[nombre_actuador]. Si el actuador ya existe en el dict, el <li> se agrega a su lista; si no, se crea una nueva entrada.

¿Por qué no emitir directamente?
Si _asignacion() emitiera HTML, el resultado de tres asignaciones consecutivas del mismo actuador sería tres bloques <div> separados. Con el acumulador, esas tres asignaciones producen un único <div> con tres <li>. Esta es la técnica que permite implementar el agrupamiento sin AST.
foco_sala.estado = ON → acum["foco_sala"] = ["<li>Estado: ON</li>"] foco_sala.brillo = 80% → acum["foco_sala"] = ["<li>Estado: ON</li>", "<li>Brillo: 80%</li>"] aire_ac.estado = OFF → acum["aire_ac"] = ["<li>Estado: OFF</li>"]
método _valor_asig() / _valor_literal() Consumen el token del lado derecho de = o de una comparación ›

_valor_asig(): acepta TIPOS_VALOR + CADENA + EMAIL. Usado en asignaciones donde el valor puede ser texto o correo.

_valor_literal(): acepta solo TIPOS_VALOR (sin CADENA ni EMAIL). Usado en condiciones donde comparar con texto o email no tiene sentido semántico.
