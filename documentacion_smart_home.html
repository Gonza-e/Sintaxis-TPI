clase TT — Token Types

classTTNamespace de todas las constantes de tipo de token›

Clase sin instancias que agrupa todas las constantes de tipo como atributos de clase. Cada atributo es un string igual a su nombre (TT.WHEN == "WHEN"). No se importa ningún módulo enum.

Novedad v4: incluye dos sets de clase TT.SENSORES y TT.ACTUADORES que agrupan los tipos de los dispositivos individuales. El lexer los usa en _actualizar_ctx() y el parser en sus validaciones semánticas.

TT.SENSORES = { SENSOR_TEMP, SENSOR_HUMEDAD, SENSOR_LUZ, SENSOR_MOVIMIENTO, SENSOR_HUMO } TT.ACTUADORES = { ACT_FOCO, ACT_AIRE, ACT_PERSIANA, ACT_CERRADURA, ACT_RELOJ, ACT_ALTAVOZ, ACT_ALARMA }

Tablas de clasificación

dictPALABRAS_RESERVADASMapeo nombre→TT, claves en UPPER, búsqueda case-insensitive›

Diccionario de 19 entradas. La búsqueda en _clasificar() se hace con palabra.upper() antes de consultar, por lo que when, When y WHEN encuentran la misma clave "WHEN". El valor del token siempre se devuelve en UPPER.

tuplePREFIJOS_SENSOR / PREFIJOS_ACTUADORCada elemento es (prefijo, TT.*) — novedad v4: tipos individuales›

En v3 existían listas separadas de prefijos de sensores y actuadores que devolvían los tipos genéricos TT.SENSOR y TT.ACTUADOR. En v4 cada entrada devuelve su tipo específico: ("sensor_temp", TT.SENSOR_TEMP), ("foco_", TT.ACT_FOCO), etc. Esto permite al parser PLY tener reglas gramaticales individuales para cada dispositivo.

PREFIJOS_SENSOR = ( ("sensor_movimiento", TT.SENSOR_MOVIMIENTO), ← más largo primero ("sensor_humedad", TT.SENSOR_HUMEDAD), ("sensor_humo", TT.SENSOR_HUMO), ("sensor_temp", TT.SENSOR_TEMP), ← evita que "sensor_temp_ext" matchee aquí ("sensor_luz", TT.SENSOR_LUZ), )
dictATRIBUTOS_CONOCIDOS / TABLA_RANGOSClasificación de atributos y rangos específicos por dispositivo›

ATRIBUTOS_CONOCIDOS: mapeo nombre_lower → TT.ATTR_*. Solo se consulta cuando el token anterior fue PUNTO, habilitando tokens tipados como ATTR_BRILLO.

TABLA_RANGOS: mapeo (tipo_dispositivo_TT, tipo_atributo_TT) → (tipo_esperado, rango, descripcion). En v4 las claves usan los nuevos tipos individuales (TT.SENSOR_TEMP, TT.ACT_FOCO) en vez del genérico anterior.

★ _distancia_edicion() y _palabra_reservada_similar()

nueva_distancia_edicion(a, b)Distancia de Levenshtein implementada a mano, sin librerías›

Calcula el mínimo de operaciones (inserción, eliminación, sustitución de un carácter) para convertir a en b. Implementada con programación dinámica (dos filas en vez de la matriz completa para ahorrar memoria).

ParámetroTipoDescripción
astrPrimera cadena (en nuestro caso, la palabra que el usuario escribió, normalizada a UPPER)
bstrSegunda cadena (la palabra reservada a comparar)
Devuelve
int — número de operaciones mínimas. 0 = idénticas, 1 = difieren en 1 carácter
_distancia_edicion("OFFF", "OFF") → 1 (una inserción de F sobrante) _distancia_edicion("WHEM", "WHEN") → 1 (sustitución M→N) _distancia_edicion("WHEN", "WHEN") → 0 (idénticas)
nueva_palabra_reservada_similar(palabra_upper)Busca la palabra reservada más parecida a una distancia de 1›

Itera las 19 palabras reservadas. Para cada una, si la diferencia de longitud es mayor a 2 se salta (optimización que evita comparar "ON" con "BLANCO"). Usa _distancia_edicion() y devuelve la que esté a distancia exactamente 1.

ParámetroTipoDescripción
palabra_upperstrLa palabra ya normalizada a UPPER que escribió el usuario
Devuelve
str | None — la palabra reservada sugerida, o None si no hay ninguna a distancia 1
¿Por qué umbral de 1? Con umbral 2 se generarían demasiados falsos positivos: "IF" estaría a distancia 2 de "IFF", pero también de "OFF", generando sugerencias erróneas. Distancia 1 captura el error más común (una letra extra, faltante o cambiada) sin ruido.

Funciones de reconocimiento _leer_*()

Todas siguen el mismo contrato: reciben (linea, pos) y retornan (valor_str, nuevo_pos) si hay coincidencia, o None si no. Este contrato uniforme permite que _siguiente_token() las pruebe en cascada.

func_leer_comentario(linea, pos)Reconoce // hasta fin de línea›

Verifica que los dos caracteres en pos y pos+1 sean //. Si es así, devuelve el resto de la línea completo y len(linea) como nueva posición, indicando que no hay más tokens en esta línea.

ParámetroTipoDescripción
lineastrTexto completo de la línea actual
posintPosición actual del cursor dentro de la línea (0-based)
func_leer_cadena(linea, pos)Reconoce texto entre comillas — acepta ASCII y tipográficas›

Verifica que el carácter en pos sea una comilla de apertura reconocida y avanza buscando la comilla de cierre. Acepta tres variantes gracias a las constantes _COMILLAS_APERTURA y _COMILLAS_CIERRE.

_COMILLAS_APERTURA = ('"', '\u201c') → " y " (ASCII + tipográfica izquierda) _COMILLAS_CIERRE = ('"', '\u201d') → " y " (ASCII + tipográfica derecha)
func_leer_numero(linea, pos)Lee un número entero o decimal con signo negativo opcional›

Función base compartida por _leer_temperatura(), _leer_con_sufijo() y el paso 13-14 de _siguiente_token(). Reconoce: [-] dígitos+ [. dígitos+].

Devuelve
(num_str, nuevo_pos) o None si no hay dígitos en pos
func_extraer_numero(s)Extrae el float del inicio de un string de token ya reconocido›

A diferencia de _leer_numero() que trabaja sobre la línea original, esta función recibe un token ya capturado (como "25°C", "80%") y extrae solo la parte numérica inicial para comparar con rangos.

_extraer_numero("25°C") → 25.0 _extraer_numero("-5.5°C") → -5.5 _extraer_numero("600lux") → 600.0
func_leer_temperatura(linea, pos)Reconoce [-]número°C o [-]número°c›

Llama a _leer_numero() y luego verifica exactamente el símbolo ° (U+00B0) seguido de C o c. Si el número tiene sufijo sospechoso (ej. °F), retorna None para que _detectar_invalido() genere el error específico.

func_leer_con_sufijo(linea, pos, sufijo)Lee número positivo + sufijo exacto (lux, %, s, m, h)›

Función genérica para reconocer PORCENTAJE (%), ILUMINANCIA (lux) y TIEMPO (s, m, h). No acepta números negativos. Verifica que después del sufijo no haya más letras.

ParámetroTipoDescripción
lineastrTexto de la línea actual
posintPosición del cursor
sufijostrEl sufijo exacto a buscar: 'lux', '%', 's', 'm' o 'h'
func_leer_tiempo(linea, pos)Prueba los tres sufijos de tiempo en orden›

Llama a _leer_con_sufijo() con 's', 'm' y 'h' en ese orden. Retorna el primero que coincida, o None si ninguno lo hace.

func_leer_hora(linea, pos)Reconoce HH:MM en formato 24h con validación de rango›

Lee exactamente 5 caracteres, verifica estructura DD:DD, y valida HH en 0-23 y MM en 0-59. Si algún valor está fuera de rango, retorna None para que _detectar_invalido() genere el error específico con el campo fuera de rango.

"22:00" → ("22:00", pos+5) "25:00" → None ← hora 25 inválida → _detectar_invalido() reporta el error "12:74" → None ← minutos 74 inválidos
func_leer_fecha(linea, pos)Reconoce DD/MM/AAAA con validación de día, mes y año›

Parsea las tres partes: día (1-31), mes (1-12), año (1900-2099). Si algún rango falla, retorna None para que _detectar_invalido() indique exactamente qué campo está mal.

"21/04/2026" → ("21/04/2026", pos+10) "41/10/2005" → None ← día 41 inválido "32/13/2026" → None ← día 32 inválido (mes 13 ni llega a evaluarse)
func_leer_email(linea, pos)Reconoce usuario@dominio.ext con validación estricta de puntos›

Recorre el dominio segmento a segmento verificando que no haya puntos consecutivos (..), punto al inicio del dominio, ni segmento vacío. La extensión (último segmento) debe tener 2-4 letras puras.

"felipe@smart-home.com.ar" → válido "gonza@gmail...com.ar" → None ← puntos consecutivos → mensaje específico "user@.dom.com" → None ← punto al inicio del dominio
nueva_tiene_sufijo_sospechoso(linea, pos)Impide que un literal mal escrito se acepte como número suelto›

Se llama justo antes de aceptar un número como ENTERO o FLOTANTE. Si inmediatamente después del número hay algo sospechoso, retorna True para que _siguiente_token() devuelva None en lugar de aceptar el número y dejar el sufijo huérfano.

ParámetroTipoDescripción
lineastrTexto de la línea actual
posintPosición justo después del número ya leído
Devuelve
bool — True si el carácter en pos sugiere un token mal formado
Caracteres que dan True: letra → "500luxx" (unidad mal escrita) ° → "25°F" (grado sin C) %+texto → "80%abc" (porcentaje con sufijo) : → "25:99" (hora inválida, ya falló _leer_hora) / → "32/13" (fecha inválida, ya falló _leer_fecha)
func_leer_identificador(linea, pos)Lee letra o _ seguido de letras, dígitos o _›

Verifica que el primer carácter sea letra o _, luego avanza mientras los caracteres sean alfanuméricos o guion bajo. El resultado se pasa a _clasificar() para determinar si es reservada, sensor, actuador, atributo o identificador.

Caso especial de email fallido: si justo después del identificador leído hay un @, _siguiente_token() retorna None en lugar de aceptar el identificador, para que _detectar_invalido() genere el mensaje de email inválido.

ParámetroTipoDescripción

métodotokenizar_linea(linea, num_linea=1)Tokeniza una sola línea sin reiniciar el contexto›

Variante para el modo interactivo. No reinicia las variables de contexto entre llamadas, de modo que instrucciones distribuidas en varias líneas mantienen coherencia dispositivo→atributo.

método_procesar_linea(linea, num_linea, tokens)Bucle principal por línea — llama a _siguiente_token y gestiona errores›

Bucle while pos < n que avanza carácter a carácter saltando blancos. Para cada posición prueba _siguiente_token(). Si retorna None, intenta _detectar_invalido() para un error específico; si también falla, emite el error genérico "carácter no reconocido".

Caso especial: si el tipo retornado por _siguiente_token() es TT.ERROR_LEX (palabra parecida a reservada), el método registra el error y emite el token ERROR_LEX sin actualizar contexto.

★ clave_siguiente_token(linea, pos, num_linea, col)Orquesta todas las funciones _leer_*() en orden de precedencia›

Método central del lexer. Prueba cada tipo de token en orden de precedencia y retorna al primer éxito. Si ninguno coincide, retorna None para que el llamador pruebe _detectar_invalido().

Orden de precedencia: 1. _leer_comentario() //... 2. _leer_cadena() "texto" o "texto" (tipográficas) 3. _leer_email() user@dom.ext (si hay @ adelante) 4. _leer_temperatura() 25°C, -5°C 5. _leer_con_sufijo('lux') 600lux 6. _leer_con_sufijo('%') 80% 7. _leer_tiempo() 30m, 10s, 1h 8. _leer_fecha() 21/04/2026 9. _leer_hora() 22:00 10. operadores dobles == != >= <= 11. operadores simples > < = 12. punto . 13. _leer_numero() → ENTERO/FLOTANTE (solo si _tiene_sufijo_sospechoso() == False) 14. _leer_identificador() → _clasificar()
★ clave_clasificar(palabra)Asigna la categoría correcta a un identificador crudo — 6 niveles de prioridad›

Recibe el texto crudo del identificador y devuelve siempre (tipo, valor). En v4 devuelve tipos individuales de sensor y actuador en lugar de los genéricos de v3.

1. upper in PALABRAS_RESERVADAS → TT.WHEN, TT.AND, TT.AZUL... (case-insensitive: "when".upper() = "WHEN") 2. lower.startswith(PREFIJO_SENSOR) → TT.SENSOR_TEMP, TT.SENSOR_LUZ... 3. lower.startswith(PREFIJO_ACT) → TT.ACT_FOCO, TT.ACT_AIRE... 4. _ultimo_tipo == PUNTO → TT.ATTR_BRILLO, TT.ATTR_MODO... 5. _palabra_reservada_similar(upper) → TT.ERROR_LEX + mensaje de sugerencia 6. fallback → TT.IDENTIFICADOR
método_actualizar_ctx(tok)Mantiene el contexto dispositivo→atributo para _validar_rango()›

Actualiza _ctx_tipo y _ctx_attr_tipo según el token recién emitido. En v4 usa los sets TT.SENSORES y TT.ACTUADORES para reconocer cualquier tipo individual en vez de comparar contra los dos strings genéricos.

si tok.tipo in TT.SENSORES: _ctx_tipo = tok.tipo, _ctx_attr_tipo = None si tok.tipo in TT.ACTUADORES: _ctx_tipo = tok.tipo, _ctx_attr_tipo = None si tok.tipo in ATRIBUTOS: _ctx_attr_tipo = tok.tipo si tok.tipo in (WHEN,IF,...): _ctx_tipo = None, _ctx_attr_tipo = None
★ clave_validar_rango(tok)Dos niveles: rango absoluto (siempre) + rango específico (con contexto)›

Valida el rango de literales TEMPERATURA, PORCENTAJE e ILUMINANCIA en dos etapas:

Nivel 1 — Rango ABSOLUTO (siempre se ejecuta, haya o no contexto): si el valor excede el rango más amplio posible para ese tipo de unidad en todo el lenguaje, es imposible que sea válido para ningún sensor/atributo. Se reporta como ERROR LÉXICO y se corta. Ej.: 2500lux, 300%, 200°C → error léxico sin importar si hay un sensor adelante o no.
Nivel 2 — Rango ESPECÍFICO (solo si hay contexto de dispositivo): si el valor pasó el rango absoluto pero excede el rango particular del sensor/atributo en contexto, se reporta como ADVERTENCIA. Ej.: 35°C es válido en general pero excede los 16-30°C de aire_.temp_obj.
_validar_rango(Token(TEMPERATURA, "200°C", ...)) → num = 200.0 → rango_abs = (-10, 50) → 200 > 50 → ERROR LÉXICO (sin contexto) _validar_rango(Token(TEMPERATURA, "35°C", ...)) [contexto: aire_.temp_obj] → num = 35.0 → rango_abs = (-10, 50) → 35 ≤ 50 → pasa nivel 1 → rango_esp = (16, 30) → 35 > 30 → ADVERTENCIA
Parser PLY

Cómo funciona PLY en este proyecto

PLY (Python Lex-Yacc) se usa únicamente para el análisis sintáctico. El análisis léxico lo sigue haciendo el lexer manual. El análisis LALR(1) construye la tabla de transiciones a partir de las funciones p_*, donde el docstring ES la regla de producción y el cuerpo ES la acción semántica.

Cómo PLY inicializa el parser yacc.yacc() ├─ recorre el módulo buscando funciones que empiecen con p_ ├─ por cada una, LEE el docstring como texto de gramática ├─ junta todas las reglas → gramática completa ├─ construye la tabla LALR(1) (autómata de estados) └─ asocia cada regla con su función Python para ejecutarla en la reducción Cómo se ejecuta una reducción PLY lee tokens del LexerAdaptador.token() acumula en la pila hasta encontrar el patrón de una regla llama a la función p_xxx(p) con: p[0] = símbolo izquierdo (donde se guarda el resultado) p[1] = primer símbolo derecho ← ya es el Token de lexer_v4 p[2] = segundo símbolo derecho ... NOTA CRÍTICA: en las reglas p_*, p[i] es _PLYToken.value = Token de lexer_v4 En p_error, p es el _PLYToken CRUDO → hay que usar p.value para llegar al Token

Diccionario que para cada combinación (TT.ACT_*, TT.ATTR_*) define qué tipos de valor acepta, si es de solo lectura, y el texto del error. En v4 las claves usan los tipos individuales de actuador.

Parser PLY

clase LexerAdaptador

classLexerAdaptadorConvierte List[Token] al protocolo que espera PLY›

PLY espera un objeto con método token() que devuelva objetos con .type, .value, .lineno, .lexpos. El adaptador recibe la lista de tokens del lexer manual y los sirve uno a uno en ese formato.

adaptador.token() devuelve: _PLYToken.type = tok.tipo ← string que PLY usa para emparejar reglas _PLYToken.value = tok ← Token completo de lexer_v4 _PLYToken.lineno = tok.linea _PLYToken.lexpos = tok.col En las reglas p_*: PLY extrae .value automáticamente → p[i] ya es el Token de lexer_v4 → p[i].tipo, p[i].valor, p[i].linea, p[i].col En p_error: p es el _PLYToken CRUDO → hay que usar p.value para llegar al Token

El adaptador filtra los tipos EOF, COMENTARIO, DESCONOCIDO y ERROR_LEX antes de servir los tokens, para que estos no lleguen al parser y no generen errores sintácticos adicionales.

Parser PLY

clase _PLYToken

class_PLYTokenObjeto mínimo compatible con el protocolo de token de PLY — sin __slots__›

Clase vacía (solo pass) sin ningún atributo declarado. El adaptador le asigna .type, .value, .lineno y .lexpos dinámicamente.

¿Por qué no usar __slots__? Cuando PLY detecta un error sintáctico, ejecuta internamente errtoken.lexer = lexer sobre el token que lo causó. Si la clase tiene __slots__ sin incluir 'lexer', esa asignación lanza: AttributeError: '_PLYToken' object has no attribute 'lexer' and no __dict__. La solución es usar una clase normal (con __dict__) que permita asignaciones dinámicas.

Parser PLY

Reglas gramaticales p_*()

En PLY, el docstring de la función ES la regla de producción (PLY lo parsea en tiempo de ejecución) y el cuerpo de la función ES la acción semántica que se ejecuta cuando esa regla se reduce. El parámetro p es una lista-like donde p[0] es el símbolo izquierdo y p[1..n] son los símbolos derechos.

reglap_programa / p_instruccion_lista_multi / p_instruccion_lista_uno / p_instruccionEl esqueleto del programa — cómo se acumulan las instrucciones›

instruccion_lista es una regla recursiva por izquierda que acumula instrucciones una a una, igual que una lista enlazada. Cada vez que PLY termina de reconocer una instrucción completa (un WHEN...END, un EVERY...END, etc.), la "agrega" a la lista acumulada.

instruccion_lista : instruccion_lista instruccion ← "la lista que ya teníamos + una más" | instruccion ← "la primera instrucción" instruccion : bloque_when | bloque_every | condicional | asignacion_suelta Acción semántica: pass (el trabajo ya lo hicieron las reglas más específicas)
¿Por qué pass? Estas reglas no necesitan hacer nada porque la traducción a HTML ocurre dentro de las reglas más específicas (p_bloque_when, p_asignacion, etc.) en el momento en que cada parte se reconoce.
reglap_bloque_when_open / p_bloque_whenWHEN condicion DO → ... → END›

Se divide en dos reglas porque PLY no permite acciones intermedias dentro de una producción. bloque_when_open se reduce justo al reconocer DO, antes de procesar el cuerpo. Eso permite emitir la cabecera HTML antes de que las asignaciones del cuerpo anoten en el acumulador.

Parámetro p[]Qué es
p[1]Token(WHEN)
p[2]Tupla (texto, sensores) devuelta por las reglas de condicion
p[3]Token(DO)
p_bloque_when_open: acción semántica texto, sensores = p[2] emit <div> + <h2>Cuando: {texto}</h2> para cada sensor: bloque_sensor() → <div verde> push_acum() ← abre el nivel de este cuerpo p[0] = ind ← devuelve la indentación para usarla en p_bloque_when p_bloque_when: acción semántica volcar(ind+" ") ← emite bloques grises de actuadores pendientes pop_acum() emit </div>
reglap_bloque_every_open / p_bloque_everyEVERY TIEMPO DO → ... → END›

Igual estructura que WHEN pero más simple: en vez de parsear una condición, solo consume el token TIEMPO directamente.

Parámetro p[]Qué es
p[2]Token(TIEMPO) — el valor del intervalo, ej. "30m"
★ clavep_cuerpo_multi / p_cuerpo_uno / p_accion_asig / p_accion_condEl cuerpo acumula acciones — igual que instruccion_lista pero dentro de un bloque›

El cuerpo es recursivo por izquierda igual que instruccion_lista. Cada vez que PLY reduce una accion (asignación o condicional anidado), la "agrega" al cuerpo acumulado. Las acciones semánticas reales están en p_asignacion y p_if_inner_open, no aquí.

cuerpo : cuerpo accion ← "más acciones a las que ya tenía" | accion ← "primera acción del cuerpo" accion : asignacion ← anota en el acumulador | condicional_inner ← vuelca actuadores + emite nuevo div
reglap_if_open / p_condicional_then / p_condicional_then_elseIF condicion THEN ... [ELSE ...] END a nivel raíz›

p_if_open se reduce justo al reconocer THEN, permitiendo emitir el encabezado HTML antes del cuerpo. Usa <h2> (nivel raíz). El ELSE es opcional.

Parámetro p[]Qué es
p[1]Token(IF)
p[2]Tupla (texto, sensores) de las reglas de condicion
p[3]Token(THEN)
reglap_else_openELSE — vuelca actuadores, cierra div THEN, abre div ELSE›

Al reconocer la palabra ELSE, antes de empezar a procesar el cuerpo del ELSE hay que: (1) volcar los actuadores pendientes del cuerpo THEN, (2) cerrar su <div>, y (3) abrir un nuevo <div> con el título "En caso contrario". Todo esto en la acción semántica de p_else_open.

reglap_if_inner_open / p_condicional_inner_then / p_else_inner_openIF anidado dentro de un cuerpo (usa h3 en vez de h2)›

Reglas equivalentes a las del IF raíz pero para IFs que aparecen dentro de un WHEN o EVERY. La diferencia principal es que p_if_inner_open llama a _st.volcar() antes de emitir el encabezado del IF, para que los actuadores que estaban antes del IF aparezcan primero en el HTML.

p_if_inner_open: acción semántica volcar(ind) ← vaciar actuadores ANTES del IF anidado (orden visual) emit <div> + <h3>Si: {texto}</h3> para cada sensor: bloque_sensor() push_acum()
★ clavep_asignacion / p_asignacion_sueltaactuador.attr = valor — valida y anota en memoria sin emitir HTML›

p_asignacion (dentro de un cuerpo) valida con chk_attr() y anota el <li> en el acumulador del actuador. NO emite HTML directamente: el bloque gris se emite después cuando se llama a volcar().

p_asignacion_suelta (nivel raíz, sin WHEN/EVERY contenedor) sí emite el bloque gris inmediatamente.

Bug histórico resuelto: antes de esta corrección, p_error hacía tok = p y luego tok.linea, lo que generaba AttributeError: '_PLYToken' object has no attribute 'linea' en cualquier error sintáctico.
def p_error(p): if p is None: # EOF inesperado — el archivo terminó antes de cerrar un bloque _st.errores_sint.append("Error sintactico | Fin de archivo inesperado") else: tok = p.value # ← p es _PLYToken, .value es el Token de lexer_v4 _st.errores_sint.append( f"Error sintactico | Linea {tok.linea:3d}, Col {tok.col:3d} | ..." )
