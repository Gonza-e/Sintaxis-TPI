clase TT — Token Types

class TT Namespace de constantes de tipo de token — sin cambios en estructura ›

Clase sin instancias que actúa como enumeración de cadenas. Cada atributo es una cadena igual a su nombre (TT.WHEN == "WHEN"). No se instancia nunca; solo agrupa y nombra las constantes.

Cambio v3: en v2 las claves de PALABRAS_RESERVADAS estaban en minúsculas ("when") y los valores de TT en mayúsculas ("WHEN"). En v3 ambos están en MAYÚSCULAS, lo que hace la clasificación directa: palabra in PALABRAS_RESERVADAS sin necesidad de normalizar.

Agrupa los tokens en: palabras reservadas, operadores lógicos, booleanos, valores discretos, dispositivos, atributos tipados, literales con unidad, literales simples, operadores y especiales.

Tablas de clasificación

dict PALABRAS_RESERVADAS Cambio v3: claves ahora en MAYÚSCULAS ›

Diccionario que mapea nombre de palabra reservada → constante TT. La búsqueda es O(1) y determina si un identificador es una palabra del lenguaje.

Cambio v3: las claves pasaron de minúsculas a MAYÚSCULAS. Esto hace la clasificación case-sensitive sin necesidad de normalizar: si el usuario escribe when, la consulta falla porque "when" no existe como clave, y el token se convierte en IDENTIFICADOR.
v2: { "when": TT.WHEN, "if": TT.IF, "and": TT.AND, ... } v3: { "WHEN": TT.WHEN, "IF": TT.IF, "AND": TT.AND, ... }
tuple / dict PREFIJOS_SENSOR / PREFIJOS_ACTUADOR / ATRIBUTOS_CONOCIDOS / TABLA_RANGOS Sin cambios en v3 ›

PREFIJOS_SENSOR / PREFIJOS_ACTUADOR: tuplas ordenadas de mayor a menor longitud de prefijo, garantizando que el prefijo más específico se encuentre primero. Sensores y actuadores siguen siendo case-insensitive porque son nombres de dispositivos dados por el usuario, no palabras del lenguaje.

ATRIBUTOS_CONOCIDOS: diccionario nombre_lower → TT.ATTR_* consultado solo cuando el token anterior fue PUNTO. Permite al parser recibir tokens específicos como ATTR_BRILLO.

TABLA_RANGOS: diccionario (prefijo, TT.ATTR_*) → (tipo_esperado, rango, desc). Consultado por _validar_rango() para emitir advertencias semánticas tempranas de valores fuera de rango.

★ Helpers de caracteres — nuevas en v3

Cinco funciones que reemplazan las clases de caracteres que antes definía el motor de expresiones regulares (\w, \d, \s, etc.). Son el bloque más primitivo sobre el que se construyen todas las funciones de reconocimiento.

nueva _es_letra(c) True si c es letra a-z o A-Z ›

Verifica si un carácter pertenece al alfabeto latino básico usando comparación de rangos Unicode directa, que es más rápida que c.isalpha() y tiene la ventaja de excluir letras con tilde y ñ (que no son válidas en el lenguaje según la consigna).

('a' <= c <= 'z') or ('A' <= c <= 'Z')
¿Por qué no usar c.isalpha()? Porque isalpha() retorna True para letras como á, ñ, ü, etc. La consigna especifica que no se admiten vocales acentuadas ni ñ, así que la comparación de rango es más correcta.
nueva _es_digito(c) True si c es dígito 0-9 ›

Verifica si un carácter es un dígito decimal usando comparación de rango. Equivalente a c.isdigit() para dígitos ASCII, pero evita que caracteres Unicode como ² o ٣ sean reconocidos como dígitos.

'0' <= c <= '9'
nueva _es_alfanum_o_guion(c) True si c puede formar parte de un identificador ›

Combina _es_letra, _es_digito y la comprobación de guion bajo. Es la condición que determina hasta dónde puede extenderse un identificador una vez que comenzó. Usada en el bucle de _leer_identificador().

_es_letra(c) or _es_digito(c) or c == '_'
nueva _es_email_char(c) True si c es válido en nombre de usuario o dominio de email ›

Define el conjunto de caracteres permitidos en las partes del email según la consigna: letras, dígitos, guion bajo, punto, signo más y signo menos. Usada tanto para validar el nombre de usuario como el dominio en _leer_email().

_es_letra(c) or _es_digito(c) or c in ('_', '.', '+', '-')
nueva _es_blanco(c) True si c es espacio, tabulación, retorno de carro o salto de línea ›

Usada en el bucle principal de _procesar_linea() para saltar los espacios entre tokens. Reemplaza la clase \s del regex.

c in (' ', '\t', '\r', '\n')

Funciones de reconocimiento — nuevas en v3

Todas siguen el mismo contrato: reciben (linea, pos) y retornan (valor_str, nuevo_pos) si hay coincidencia, o None si no. Este contrato uniforme permite que _siguiente_token() las pruebe en cascada de forma limpia.

nueva _leer_comentario(linea, pos) Reconoce // hasta el fin de la línea ›

Verifica si los dos caracteres en pos y pos+1 son ambas barras /. Si es así, retorna el resto completo de la línea como valor y la posición final como nuevo_pos, indicando que no hay más tokens en esta línea.

linea[pos] == '/' and linea[pos+1] == '/' → retorna (linea[pos:], len(linea))
nueva _leer_cadena(linea, pos) Reconoce texto entre comillas dobles "..." ›

Verifica que el primer carácter sea " y avanza buscando la comilla de cierre. Si encuentra " o el fin de línea antes de cerrar, determina si la cadena es válida o no. Una cadena sin comilla de cierre retorna None — el llamador generará un error léxico.

linea[pos] == '"' i = pos+1 mientras linea[i] != '"' y linea[i] != '\n': i++ si linea[i] == '"': retorna (linea[pos:i+1], i+1) si no: retorna None ← comilla sin cerrar
nueva _leer_numero(linea, pos) Lee un número entero o decimal con signo negativo opcional ›

Función base compartida por _leer_temperatura(), _leer_con_sufijo() y los pasos 13-14 de _siguiente_token(). Reconoce: signo - opcional → dígitos enteros → punto decimal → dígitos decimales (los dos últimos también opcionales).

[-] dígitos+ [. dígitos+] "-25.5" → ("-25.5", pos+5) "80" → ("80", pos+2) "abc" → None
Importante: esta función solo lee el número, no el sufijo. El token ENTERO o FLOTANTE se determina en _siguiente_token() chequeando si el resultado contiene '.'.
nueva _extraer_numero(s) Extrae el valor float del inicio de un string de token — reemplaza re.match ›

Usada por los helpers numéricos (_num_temperatura, _num_porcentaje, _num_iluminancia) para extraer el número de un token ya reconocido como "25°C", "80%" o "600lux". Avanza carácter a carácter leyendo signo, parte entera y parte decimal, y retorna el float correspondiente.

Lo que reemplaza: en v2 esta operación se hacía con re.match(r"(-?\d+(?:\.\d+)?)", s).group(1). En v3 la misma lógica se implementa con un índice i que avanza mientras encuentra dígitos.
"25°C" → 25.0 "-5.5°C" → -5.5 "80%" → 80.0 "600lux" → 600.0
nueva _leer_temperatura(linea, pos) Reconoce [-]número°C o [-]número°c ›

Llama a _leer_numero() para obtener la parte numérica y luego verifica que le sigan exactamente los caracteres ° (Unicode U+00B0) y C o c. Si alguno de los dos falta, retorna None.

"26°C" → ("26°C", pos+4) "-5°c" → ("-5°c", pos+4) "26°F" → None ← no es Celsius "26C" → None ← falta el símbolo °
nueva _leer_con_sufijo(linea, pos, sufijo) Lee número positivo seguido exactamente de un sufijo dado ›

Función genérica para reconocer PORCENTAJE (%), ILUMINANCIA (lux) y las unidades de tiempo (s, m, h). No acepta números negativos. Verifica además que después del sufijo no haya más letras, para evitar que 5min sea reconocido como 5m + in.

_leer_con_sufijo(linea, pos, '%'): "80%" → ("80%", pos+3) "100%" → ("100%", pos+4) "-5%" → None ← no acepta negativo "5min" → None ← 'm' seguido de 'i' (más letras) _leer_con_sufijo(linea, pos, 'lux'): "600lux" → ("600lux", pos+6)
nueva _leer_tiempo(linea, pos) Reconoce número seguido de s, m o h ›

Prueba las tres unidades de tiempo en orden llamando a _leer_con_sufijo() con cada una. Retorna el resultado del primero que coincida, o None si ninguno coincide.

para unidad en ('s', 'm', 'h'): res = _leer_con_sufijo(linea, pos, unidad) si res: retornar res retornar None "30m" → ("30m", pos+3) "10s" → ("10s", pos+3) "1h" → ("1h", pos+2)
nueva _leer_hora(linea, pos) Reconoce HH:MM en formato 24 horas con validación de rango ›

Lee exactamente 5 caracteres y verifica estructura DD:DD, valores HH en 0-23 y MM en 0-59. También verifica que el 6° carácter no sea un dígito (para no capturar 12:345 como 12:34).

s = linea[pos:pos+5] ← "22:00" s[2] == ':' and todos son dígitos hh = int(s[0:2]) → 0..23 mm = int(s[3:5]) → 0..59 linea[pos+5] no es dígito "22:00" → ("22:00", pos+5) "25:00" → None ← hora inválida "12:60" → None ← minutos inválidos
nueva _leer_fecha(linea, pos) Reconoce DD/MM/AAAA con validación de día, mes y año ›

La función más larga de las de reconocimiento. Parsea las tres partes de la fecha en secuencia: día (1-2 dígitos, 1-31), barra, mes (1-2 dígitos, 1-12), barra, año (exactamente 4 dígitos, 1900-2099). Verifica después del año que no haya más dígitos.

Día: 1-31 (1 o 2 dígitos) → '/' Mes: 1-12 (1 o 2 dígitos) → '/' Año: 1900-2099 (exactamente 4 dígitos) "21/04/2026" → ("21/04/2026", pos+10) "31/13/2026" → None ← mes 13 inválido "21/04/1800" → None ← año fuera de rango
Sin validación día/mes cruzada: la consigna indica que no es necesario validar que el día sea coherente con el mes (ej. 31 de febrero se acepta). Solo se valida que el día esté en 1-31 y el mes en 1-12 independientemente.
nueva _leer_email(linea, pos) Reconoce usuario@dominio.ext con extensión de 2-4 letras ›

Parsea el email en tres etapas: nombre de usuario (al menos un char válido según _es_email_char()), el @ obligatorio, y el dominio. Dentro del dominio busca el último punto usando rfind('.') y verifica que la extensión sea de 2 a 4 letras puras.

usuario: chars válidos (_es_email_char) + al menos 1 @: obligatorio dominio: chars válidos, con al menos un punto extensión: 2-4 letras (último segmento tras el último punto) "user@dom.com" → válido "f@s-h.com.ar" → válido (extensión "ar") "user@dom" → None ← sin punto en dominio "user@dom.comx" → None ← extensión > 4 letras
nueva _leer_identificador(linea, pos) Reconoce nombre: letra o _ seguido de letras, dígitos o _ ›

Verifica que el primer carácter sea letra o guion bajo y luego avanza mientras los caracteres siguientes sean alfanuméricos o guion bajo, usando _es_alfanum_o_guion(). El resultado se pasa a _clasificar() para determinar si es reservada, sensor, actuador, atributo o identificador genérico.

"WHEN" → ("WHEN", pos+4) → _clasificar → TT.WHEN "sensor_luz" → ("sensor_luz", pos+10) → _clasificar → TT.SENSOR "foco_sala" → ("foco_sala", pos+9) → _clasificar → TT.ACTUADOR "brillo" (tras .) → ("brillo", pos+6) → _clasificar → TT.ATTR_BRILLO "mi_variable" → ("mi_variable", ...) → TT.IDENTIFICADOR

método _num_temperatura(v) / _num_porcentaje(v) / _num_iluminancia(v) Extraen el float de un token literal — ahora usan _extraer_numero() ›

clase SmartHomeLexer

método tokenizar(fuente, num_linea_base=1) API principal — sin cambios en v3 ›

Divide el texto en líneas y llama a _procesar_linea() por cada una. Reinicia errores, advertencias y el contexto al inicio. Agrega un token EOF al final. No cambió en v3.

Retorna
List[Token] — todos los tokens sin blancos ni comentarios, más EOF
método tokenizar_linea(linea, num_linea=1) Modo interactivo — sin cambios en v3 ›

Variante para el modo interactivo. Procesa una sola línea sin reiniciar el contexto entre llamadas.

método _procesar_linea(linea, num_linea, tokens) Bucle principal del lexer — modificado en v3 ›

Bucle que avanza posición a posición sobre la línea. En v2 llamaba a _REGEX_MASTER.match(linea, pos); en v3 llama a _siguiente_token(linea, pos, ...). La lógica alrededor (saltar blancos, manejar None, descartar comentarios, actualizar contexto) no cambió.

Cambio v3: _REGEX_MASTER.match() → self._siguiente_token(). El resto del método es idéntico.
★ nueva _siguiente_token(linea, pos, num_linea, col) Orquesta todas las funciones _leer_*() en orden de precedencia — reemplaza _REGEX_MASTER ›

Método central del lexer v3. Intenta reconocer el próximo token probando cada tipo en orden de precedencia. Retorna (tipo, valor, nuevo_pos) al primer éxito, o None si ninguno coincide.

¿Por qué este orden? El orden importa para evitar ambigüedades. Por ejemplo: 250lux debe ser ILUMINANCIA, no ENTERO+IDENTIFICADOR. 30m debe ser TIEMPO, no ENTERO+IDENTIFICADOR. user@dom.com debe ser EMAIL, no IDENTIFICADOR. Los tipos más específicos van primero.
Orden de precedencia: 1. _leer_comentario() //... 2. _leer_cadena() "texto" 3. _leer_email() user@dom.ext 4. _leer_temperatura() 25°C, -5°C 5. _leer_con_sufijo('lux') 600lux 6. _leer_con_sufijo('%') 80% 7. _leer_tiempo() 30m, 10s, 1h 8. _leer_fecha() 21/04/2026 9. _leer_hora() 22:00 10. comparación de 2 chars == != >= <= 11. comparación de 1 char > < = 12. punto . 13. _leer_numero() → FLOTANTE 3.14 14. _leer_numero() → ENTERO 42 15. _leer_identificador() WHEN / sensor_luz / foco_ / brillo / nombre
método _clasificar(palabra) Modificado en v3 — eliminado el lower() para palabras reservadas ›

Clasifica un identificador en su categoría. Mantiene 5 niveles de prioridad pero el primero cambió en v3.

Cambio v3: En v2 se normalizaba lower = palabra.lower() y se consultaba con la clave en minúsculas. En v3 la consulta es directa con la palabra tal como vino, porque las claves de PALABRAS_RESERVADAS están en MAYÚSCULAS. Escribir when falla la consulta → IDENTIFICADOR.
v2: lower = palabra.lower() if lower in PALABRAS_RESERVADAS: ← "when" encontraba "when" return PALABRAS_RESERVADAS[lower], palabra.upper() v3: if palabra in PALABRAS_RESERVADAS: ← "when" NO encuentra nada return PALABRAS_RESERVADAS[palabra], palabra lower = palabra.lower() ← solo para sensor/actuador/atributo

Los pasos 2-5 (sensores, actuadores, atributos, genérico) siguen usando lower() porque son nombres de dispositivos definidos por el usuario, no palabras del lenguaje.

método _actualizar_ctx(tok) / _prefijo_canonico(nombre) Autómata de contexto dispositivo→atributo — sin cambios en v3 ›

_prefijo_canonico() extrae el prefijo canónico de un nombre completo ("aire_acondicionado" → "aire_"). _actualizar_ctx() actualiza las variables de estado _ctx_prefijo y _ctx_attr_tipo en función del tipo del token recién emitido, permitiendo que _validar_rango() sepa en qué contexto se encuentra el próximo literal.

método _validar_rango(tok) Advertencias semánticas tempranas — sin cambios en v3 ›

Consulta TABLA_RANGOS con la clave (prefijo_actual, attr_tipo_actual). Si existe una entrada, usa _EXTRACTOR[tok.tipo](tok.valor) para obtener el número y lo compara con el rango. Si está fuera, agrega una advertencia a self.advertencias. En v3 el extractor llama a _extraer_numero() en lugar de re.match.

Gramática con acciones semánticas

Las llaves { … } indican acciones semánticas: código Python que se ejecuta en el momento en que esa parte de la regla gramatical es reconocida. Esta notación es estándar en la teoría de compiladores (esquemas de traducción dirigidos por la sintaxis).

programa ::= { instruccion }+ EOF { _emitir_cabecera() } { _emitir_cierre() } instruccion ::= bloque_when | bloque_every | condicional | asignacion_suelta bloque_when ::= WHEN condicion DO { _emit(<div seccion>) + _emit(<h2>Cuando:…</h2>) } { para cada sensor: _html_sensor() } cuerpo END { _emit(</div>) } bloque_every::= EVERY TIEMPO DO { _emit(<div seccion>) + _emit(<h2>Cada: TIEMPO</h2>) } cuerpo END { _emit(</div>) } cuerpo ::= accion+ { mantiene OrderedDict { actuador: [<li>…] } } { al llegar a END/ELSE: _volcar_actuadores() } accion ::= asignacion | condicional condicional ::= IF condicion THEN { _emit(<div seccion>) + _emit(<h3>Si:…</h3>) } { para cada sensor: _html_sensor() } cuerpo [ ELSE { _emit(</div>) + _emit(<div>) + _emit(<h3>En caso contrario</h3>) } cuerpo ] END { _emit(</div>) } asignacion ::= ACTUADOR PUNTO ATTR OP_ASIG valor_asig { acum[actuador].append(_html_item()) ← NO emite todavía } condicion ::= expr_cond ( (AND|OR) expr_cond )* { retorna (texto_legible, sensores) ← NO emite HTML } expr_cond ::= NOT expr_cond | TRUE | FALSE | ON | OFF | SENSOR op_cmp valor_literal | ACTUADOR PUNTO ATTR op_cmp valor_literal

set TIPOS_OP_CMP / TIPOS_ATTR / TIPOS_VALOR Sets de constantes TT para lookahead en _es() y _consumir() ›

TIPOS_OP_CMP: agrupa los seis operadores de comparación (== != > < >= <=). Se usa con _consumir(*TIPOS_OP_CMP) para aceptar cualquiera de ellos en una regla, en lugar de listarlos uno por uno.

TIPOS_ATTR: todos los tokens ATTR_* posibles. Usado en _asignacion() y _expr_cond() para consumir cualquier atributo reconocido por el lexer.

TIPOS_VALOR: todos los tokens que pueden aparecer como valor (literales con unidad, booleanos, discretos, números, identificadores). Usado en _valor_asig() y _valor_literal().

TIPOS_OP_CMP = { TT.OP_EQ, TT.OP_NEQ, TT.OP_GT, TT.OP_LT, TT.OP_GTE, TT.OP_LTE } TIPOS_ATTR = { TT.ATTR_ESTADO, TT.ATTR_BRILLO, TT.ATTR_COLOR, TT.ATTR_MODO, TT.ATTR_TEMP_OBJ, TT.ATTR_TEMP_ACT, TT.ATTR_POSICION, ... } TIPOS_VALOR = { TT.TEMPERATURA, TT.PORCENTAJE, TT.ON, TT.OFF, TT.TRUE, TT.FRIO, TT.CALOR, TT.ENTERO, TT.FLOTANTE, ... }
dict _OP_TEXTO / _LABEL_SENSOR / _UNIDAD_SENSOR Tablas de traducción para la generación HTML ›

_OP_TEXTO: mapea cada tipo de operador TT a su símbolo legible. Usado al construir el texto del título de sección HTML (TT.OP_GTE → ">=").

_LABEL_SENSOR: mapea el nombre técnico del sensor ("sensor_temp") a un label legible ("Temperatura") para mostrar en el <h2> del bloque verde.

_UNIDAD_SENSOR: mapea el nombre del sensor a su unidad de medida ("sensor_temp" → "°C") para incluirla en el texto del bloque HTML.

_OP_TEXTO = { TT.OP_EQ: "==", TT.OP_GT: ">", TT.OP_GTE: ">=", ... } _LABEL_SENSOR = { "sensor_temp": "Temperatura", "sensor_luz": "Iluminancia", ... } _UNIDAD_SENSOR = { "sensor_temp": "°C", "sensor_humedad": "%", "sensor_luz": "lux" }
str const _CSS_SENSOR / _CSS_ACTUADOR / _CSS_SECCION Estilos CSS inline definidos en la consigna (sección 5) ›

Atributos de clase de ParserHTML que centralizan los estilos CSS requeridos por la consigna para no repetirlos en cada llamada de emisión.

_CSS_SENSOR = "border:1px solid green; padding:20px; margin:10px 0;" _CSS_ACTUADOR = "border:1px solid gray; padding:20px; margin:10px 0;" _CSS_SECCION = "margin:20px 0; padding:12px 16px; border-left:3px solid #4a90d9; ..."

Parser

clase ParserHTML

Parser LL(1) descendente recursivo. Cada método corresponde a una regla de producción de la gramática. El HTML se genera durante el análisis sin construir ningún árbol intermedio.

método _actual() / _es(*tipos) Lookahead de 1 símbolo — inspecciona sin consumir ›

_actual() retorna el token en self._tokens[self._pos] sin avanzar el cursor. _es(*tipos) retorna True si el tipo del token actual está en la lista de tipos dados. Son la base del lookahead LL(1): permiten decidir qué regla gramatical aplicar sin consumir ningún token.

if self._es(TT.WHEN): # ¿el próximo es WHEN? self._bloque_when(nivel) # sí → aplicar esa regla if self._es(TT.END, TT.ELSE, TT.EOF): # varios tipos a la vez break
método _consumir(*tipos) Verifica tipo, avanza el cursor y retorna el token — o lanza ErrorSintactico ›

Operación fundamental del parser. Si el token actual es de uno de los tipos esperados, avanza self._pos y retorna el token. Si no, construye un mensaje de error con los tipos esperados y lanza ErrorSintactico. Es el método más invocado del parser.

tok_when = self._consumir(TT.WHEN) # solo acepta WHEN tok_tiempo = self._consumir(TT.TIEMPO) # solo acepta TIEMPO tok_attr = self._consumir(*TIPOS_ATTR) # acepta cualquier ATTR_*
Diferencia con _avanzar(): _consumir() valida el tipo antes de avanzar. _avanzar() avanza incondicionalmente. Se usa _avanzar() solo donde el tipo ya fue verificado por _es() en la línea inmediata anterior.
método _avanzar() Avanza el cursor incondicionalmente y retorna el token actual ›

Usado donde el tipo ya fue verificado con _es(), haciendo redundante una segunda validación. Ejemplo típico:

if self._es(TT.ELSE): self._avanzar() # ← ya sabemos que es ELSE, no necesitamos _consumir if self._es(TT.AND, TT.OR): op_logic = self._avanzar().valor.upper() # capturar AND/OR sin revalidar
método _sincronizar(*tipos_seguimiento) Recuperación de pánico: descarta tokens hasta un punto seguro ›

Implementa la estrategia de recuperación de errores panic mode. Descarta tokens uno por uno hasta encontrar alguno del conjunto de seguimiento, desde donde el análisis puede retomarse con coherencia. Permite reportar múltiples errores en una sola ejecución en lugar de abortar al primero.

// En parsear(): sincronizar al inicio de la siguiente instrucción self._sincronizar(TT.WHEN, TT.EVERY, TT.IF, TT.END, TT.EOF) // En _cuerpo(): sincronizar a la siguiente acción o al cierre del bloque self._sincronizar(TT.ACTUADOR, TT.IF, TT.END, TT.ELSE, TT.EOF)
Panic mode recovery: la más simple de las estrategias de recuperación. Funciona bien para este lenguaje porque los bloques WHEN, EVERY e IF tienen delimitadores claros (END), lo que permite encontrar puntos de re-sincronización de forma confiable.
método _emit(linea) Agrega una línea al buffer de salida HTML ›

El método más simple del parser: hace self._out.append(linea). Es la única puerta de salida para producir HTML. Al final de parsear(), el buffer completo se une con "\n".join(self._out).

¿Por qué buffer y no escribir directamente al archivo? Usar una lista en memoria permite retornar el HTML como string desde parsear(). Esto facilita el testing unitario y el modo interactivo, donde se imprime en consola en lugar de escribir a disco. El archivo se escribe en modo_archivo(), no dentro del parser.
método _html_sensor(nombre, op, valor, ind) Acción semántica: emite bloque verde de sensor según la consigna ›

Genera el bloque HTML de un sensor con border:1px solid green; padding:20px tal como indica la sección 5 de la consigna. Traduce el nombre técnico del sensor a label legible usando _LABEL_SENSOR, agrega la unidad desde _UNIDAD_SENSOR, y formatea el operador con _OP_TEXTO.

_html_sensor("sensor_temp", TT.OP_GT, "26°C", " ") emite: <div style="border:1px solid green; padding:20px; margin:10px 0;"> <h2>Temperatura > 26°C°C</h2> </div>
¿Cuándo se llama? Justo después de emitir el <div> contenedor de la sección, en _bloque_when() y _condicional(). Los datos del sensor llegan de _condicion() que los recolecta sin emitir nada.
método _html_actuador(nombre, items, ind) Acción semántica: emite bloque gris completo de actuador ›

Genera el bloque HTML del actuador con border:1px solid gray; padding:20px, el nombre en <h1> y los atributos como <ul><li>, según la consigna. Recibe la lista de strings <li> ya construidos por _html_item() y los vuelca de una vez. Este método no sabe cuántos atributos hay ni cuándo llegaron — simplemente emite lo que _cuerpo() acumuló.

_html_actuador("foco_sala", ["<li>Estado: ON</li>", "<li>Brillo: 80%</li>"], " ") emite: <div style="border:1px solid gray; padding:20px; margin:10px 0;"> <h1>foco_sala</h1> <ul> <li>Estado: ON</li> <li>Brillo: 80%</li> </ul> </div>
método _html_item(attr_nombre, valor_tipo, valor_raw) Construye el string <li> — NO lo emite todavía ›

Construye y retorna el string <li> sin llamar a _emit(). El llamador (_asignacion()) lo deposita en el acumulador. Maneja tres casos según el tipo del valor:

EMAIL → <li>Email notif: <a href="mailto:user@dom.ar">Contactar a user</a></li> CADENA → <li>Mensaje: texto sin comillas dobles</li> otros → <li>Estado: ON</li>

Para EMAIL extrae el nombre de usuario con valor_raw.split("@")[0] y genera un link mailto: tal como especifica la consigna. Para CADENA elimina las comillas con strip('"').

★ clave _volcar_actuadores(acumulador, ind) Convierte el dict acumulado en bloques HTML y lo limpia ›

Itera el OrderedDict y llama a _html_actuador() por cada entrada, luego limpia el dict con .clear(). Es el momento exacto en que las asignaciones acumuladas se convierten en HTML real.

¿Por qué este método es clave? Sin él, sería imposible agrupar múltiples asignaciones del mismo actuador en una sola pasada. El patrón es: las asignaciones recuerdan en lugar de emitir, y este método es el que finalmente convierte lo recordado en HTML. Se llama en dos momentos precisos: 1. Antes de entrar a un IF anidado → para que los actuadores aparezcan ANTES de la subsección 2. Al salir del bucle de _cuerpo() → para vaciar lo que quedó pendiente al llegar a END o ELSE
acum = { "foco_sala": ["<li>Estado: ON</li>", "<li>Brillo: 80%</li>"], "aire_ac": ["<li>Modo: FRIO</li>"] } _volcar_actuadores(acum, " ") → emite bloque gris de foco_sala con 2 ítems → emite bloque gris de aire_ac con 1 ítem → acum queda {} (clear())
método parsear() Punto de entrada público — orquesta el análisis completo ›

Método público que inicia todo el proceso. Llama a _emitir_cabecera(), entra en un bucle que invoca _instruccion() hasta llegar a EOF con recuperación de pánico ante errores, y finaliza llamando a _emitir_cierre(). Retorna el HTML completo como string uniendo el buffer con "\n".join(self._out).

Retorna
str — documento HTML completo listo para escribir a disco
método _emitir_cabecera() / _emitir_cierre() Emiten el DOCTYPE, head, body de apertura y el cierre del documento ›

_emitir_cabecera() genera el encabezado HTML completo: <!DOCTYPE html>, metadatos UTF-8, viewport, título con el nombre del archivo, bloque <style> con reglas CSS básicas, y el <h1> del dashboard. Se llama una única vez al inicio de parsear().

_emitir_cierre() emite únicamente </body></html>. Se llama una única vez al final.

método _instruccion(nivel) Despacha al método correcto según el lookahead de 1 token ›

Regla: instruccion ::= bloque_when | bloque_every | condicional | asignacion_suelta. Usa _es() para mirar el token actual y despachar al método correspondiente. Si el token no es ninguno de los esperados, lanza ErrorSintactico.

WHEN → _bloque_when(nivel) EVERY → _bloque_every(nivel) IF → _condicional(nivel) ACTUADOR → acum=OrderedDict(); _asignacion(acum); _volcar_actuadores(acum, " ") otro → ErrorSintactico("Se esperaba WHEN, EVERY, IF o ACTUADOR")

El caso ACTUADOR crea su propio OrderedDict local para asignaciones sueltas a nivel raíz (sin bloque WHEN/EVERY que las contenga) y lo vuelca inmediatamente.

método _bloque_when(nivel) WHEN condicion DO cuerpo END con acciones semánticas intercaladas ›

Implementa la regla bloque_when intercalando emisión HTML. La condición se parsea antes de emitir el <div> porque necesitamos el texto y la lista de sensores para el <h2> y los bloques verdes. _condicion() retorna esos datos sin emitir nada.

consumir WHEN (texto, sensores) ← _condicion() ← NO emite, solo recolecta consumir DO _emit(<div ...>) ← abre sección _emit(<h2>Cuando: {texto}</h2>) para cada sensor: _html_sensor(...) ← bloque verde _cuerpo(nivel+1) ← acumula y vuelca actuadores consumir END _emit(</div>) ← cierra sección
método _bloque_every(nivel) EVERY TIEMPO DO cuerpo END ›

Idéntico en estructura a _bloque_when() pero más simple: el disparador es directamente el token TIEMPO (ej. 30m), sin necesidad de parsear una condición. El valor del tiempo se incluye en el <h2> de la sección.

consumir EVERY tok_tiempo ← consumir(TT.TIEMPO) consumir DO _emit(<div ...>) _emit(<h2>Cada: {tok_tiempo.valor}</h2>) _cuerpo(nivel+1) consumir END _emit(</div>)
★ clave _cuerpo(nivel) Núcleo del agrupamiento en una pasada — mantiene el OrderedDict local ›

El método más importante del parser para entender la técnica de una sola pasada. La clave está en el OrderedDict local que vive solo durante la ejecución de este método:

Ciclo de vida del acumulador:
1. Se crea vacío al entrar: acum = OrderedDict()
2. Cada _asignacion(acum) deposita un <li> en acum[nombre_actuador]
3. Si aparece un IF, primero se vuelcan los actuadores pendientes (orden visual correcto) y luego se parsea el IF
4. Al encontrar END o ELSE, se vuelcan todos los actuadores restantes
5. Al retornar, el dict se destruye automáticamente (variable local)
acum = OrderedDict() mientras NO (END | ELSE | EOF): si IF: _volcar_actuadores(acum, ind) ← vaciar ANTES del IF _condicional(nivel) si ACTUADOR: _asignacion(acum) ← acumular, NO emitir si otro: ErrorSintactico + _sincronizar() _volcar_actuadores(acum, ind) ← vaciar lo que queda
¿Por qué OrderedDict y no dict ordinario? En Python 3.7+ los dict ya mantienen orden de inserción, así que técnicamente ambos son equivalentes. Se usa OrderedDict explícitamente para comunicar que el orden de aparición de los actuadores es intencional y debe preservarse en el HTML.
método _condicional(nivel) IF condicion THEN cuerpo [ELSE cuerpo] END ›

Parsea el bloque IF completo intercalando emisión HTML. El ELSE es opcional. El parámetro nivel determina el tag del título: <h2> si es nivel raíz (dentro de WHEN/EVERY), <h3> si está anidado dentro de otro IF.

consumir IF (texto, sensores) ← _condicion() consumir THEN tag = "h3" si nivel > 0 else "h2" _emit(<div> + <{tag}>Si: {texto}</{tag}>) para cada sensor: _html_sensor() _cuerpo(nivel+1) ← cuerpo THEN con su propio acum si ELSE: _avanzar() _emit(</div>) ← cierra div THEN _emit(<div> + <{tag}>En caso contrario</{tag}>) _cuerpo(nivel+1) ← cuerpo ELSE con su propio acum consumir END _emit(</div>) ← cierra div THEN o ELSE
método _condicion() Encadena expr_cond con AND/OR — retorna datos, NO emite HTML ›

Regla: condicion ::= expr_cond ((AND|OR) expr_cond)*. Parsea la primera expresión y luego mientras el token sea AND u OR, parsea otra y concatena el texto y las listas de sensores. El HTML no se emite aquí porque en el momento de parsear la condición aún no se abrió el <div> contenedor.

Retorna
(texto: str, sensores: list[tuple(nombre, op_tipo, valor)])
"sensor_temp > 26°C AND alarma_casa.estado == OFF" → texto = "sensor_temp > 26°C AND alarma_casa.estado == OFF" → sensores = [("sensor_temp", TT.OP_GT, "26°C")] (los actuadores en condición NO se agregan a sensores)
método _expr_cond() Expresión simple: sensor op valor, actuador.attr op valor, o booleano ›

Parsea una de cuatro formas posibles y retorna (texto, sensores). Cuando el lado izquierdo es un SENSOR registra la tupla (nombre, op_tipo, valor) en la lista de sensores para que el llamador emita el bloque verde. Los actuadores en condición no generan bloque verde.

NOT expr_cond → ("NOT {texto}", sensores heredados) TRUE | FALSE | ON | OFF → ("TRUE", []) SENSOR op valor → ("sensor_luz < 250lux", [("sensor_luz", OP_LT, "250lux")]) ACTUADOR PUNTO ATTR op valor → ("aire_ac.estado == OFF", []) ← sin bloque verde
★ clave _asignacion(acum) Acumula en el dict — NO emite HTML directamente ›

Implementa la regla asignacion ::= ACTUADOR PUNTO ATTR OP_ASIG valor_asig. Consume los cinco tokens en orden, construye el string <li> con _html_item() y lo deposita en acum[nombre_actuador] sin llamar a _emit().

¿Por qué no emitir directamente? Si _asignacion() emitiera HTML, tres asignaciones consecutivas del mismo actuador producirían tres <div> separados. Con el acumulador, esas tres asignaciones producen un único <div> con tres <li>. Este es el mecanismo que hace posible el agrupamiento sin AST.
foco_sala.estado = ON → acum["foco_sala"] = ["<li>Estado: ON</li>"] foco_sala.brillo = 80% → acum["foco_sala"] = ["<li>Estado: ON</li>", "<li>Brillo: 80%</li>"] aire_ac.estado = OFF → acum["aire_ac"] = ["<li>Estado: OFF</li>"]
método _valor_asig() / _valor_literal() Consumen el token del lado derecho de = o de una comparación ›

_valor_asig(): acepta todos los tokens de TIPOS_VALOR más CADENA y EMAIL. Usado en asignaciones donde el valor puede ser texto entre comillas o una dirección de correo.

_valor_literal(): acepta solo TIPOS_VALOR (sin CADENA ni EMAIL). Usado en condiciones donde comparar con texto o email con operadores >, <, == no tiene sentido semántico.

asignacion: altavoz.mensaje = "texto" → _valor_asig() acepta CADENA asignacion: altavoz.email_notif = u@d.com → _valor_asig() acepta EMAIL condicion: sensor_temp > 26°C → _valor_literal() acepta TEMPERATURA condicion: sensor_temp > "texto" → ErrorSintactico (CADENA no en TIPOS_VALOR)
